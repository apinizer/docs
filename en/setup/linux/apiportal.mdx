title: "API Portal — Linux / VM"

This guide describes how to install Apinizer API Portal as a standalone
service on a Linux server (Ubuntu, RHEL, Debian, Oracle Linux, AlmaLinux,
Rocky, etc.). The distribution ships with a bundled Temurin OpenJDK 25
runtime — Java is **not** required on the host.

Use this installation model when:

- Kubernetes is not available or not desired in the target environment
- A single-node deployment is sufficient
- You need encrypted credentials at rest (finance / regulated environments)

## Requirements

| Component  | Requirement |
|------------|-------------|
| OS         | Linux x86-64 (glibc-based: Ubuntu 20.04+, RHEL 8+, Debian 11+, Oracle/AlmaLinux/Rocky 8+) |
| CPU        | 2 cores minimum (x86-64-v3) |
| RAM        | 2 GB minimum, 4 GB recommended |
| Disk       | 1 GB free under the install directory; additional space for logs |
| Network    | Outbound HTTPS to the Apinizer Manager REST endpoint |
| Privileges | A user with `sudo` rights — root login is **not** required |
| Java       | Not required (bundled JRE) |

## 1. Download the Package

Get the latest standalone tarball and its checksum from the Apinizer FSS:

```bash
VERSION="2026.04.2"
curl -fLO "https://packages.apinizer.com/apinizer-packages/apiportal/${VERSION}/apinizer-apiportal-${VERSION}-linux-x64.tar.gz"
curl -fLO "https://packages.apinizer.com/apinizer-packages/apiportal/${VERSION}/checksums.sha256"
```

Verify integrity:

```bash
sha256sum -c checksums.sha256
# Expected: apinizer-apiportal-<version>-linux-x64.tar.gz: OK
```

## 2. Extract to the Install Directory

The standard install location is `/opt/apinizer-portal`, but any path
your operator owns will work.

```bash
sudo mkdir -p /opt
sudo tar -xzf apinizer-apiportal-${VERSION}-linux-x64.tar.gz -C /opt/
sudo mv /opt/apinizer-apiportal-${VERSION} /opt/apinizer-portal
```

Resulting layout:

```
/opt/apinizer-portal/
├── bin/        Start, stop, install and encrypt scripts; systemd unit
├── conf/       application.env (and master.key after install)
├── lib/        Application jar + lib/jasypt/jasypt-*.jar (encryption helper)
├── runtime/    Bundled Temurin OpenJDK 25
└── logs/       Created on first start
```

## 3. Configure `application.env`

Edit the configuration file and fill in the **required** parameters:

```bash
sudo vi /opt/apinizer-portal/conf/application.env
```

```env
# REQUIRED
API_PORTAL_ID=<id-from-the-manager>
API_PORTAL_MANAGEMENT_API_BASE_URL=https://manager.example.com
API_PORTAL_MANAGEMENT_API_KEY=<api-key-from-the-manager>

# Optional
SERVER_PORT=8080
```

The `API_PORTAL_ID` and `API_PORTAL_MANAGEMENT_API_KEY` values are
generated from the Apinizer Manager UI under **Portal → API Keys**.

<Info>
At this point you may leave secrets as plaintext and proceed; encryption
is added in Step 5 and is opt-in per field.
</Info>

## 4. Run the Installer

The installer creates a dedicated `apinizer` system user (no login
shell), sets file permissions, generates a random Jasypt master key,
validates the configuration, and registers a `systemd` service.

```bash
sudo /opt/apinizer-portal/bin/apiportal-install.sh
```

Expected output:

```
[OK] User 'apinizer' created.
[OK] Permissions set.
[OK] Master key generated: /opt/apinizer-portal/conf/master.key
[OK] conf/application.env validated.
[OK] Systemd service installed and enabled.
```

The installer is **idempotent** — running it again will not regenerate
the master key (which would invalidate previously encrypted values) and
will not duplicate the user.

### Sudoers entry for non-root operators

If your security policy forbids interactive root logins, give a single
operator account the right to run only the installer:

```bash
echo 'apinizer-admin ALL=(root) NOPASSWD: /opt/apinizer-portal/bin/apiportal-install.sh' \
  | sudo tee /etc/sudoers.d/apinizer-portal
sudo chmod 440 /etc/sudoers.d/apinizer-portal
```

`apinizer-admin` may then run `sudo /opt/apinizer-portal/bin/apiportal-install.sh`
without needing the root password or full sudo privileges.

## 5. Encrypt Sensitive Values (Optional, Recommended)

Sensitive credentials in `application.env` (API keys, keystore
passwords) can be encrypted at rest using the Jasypt master key
generated by the installer. The master key file
(`conf/master.key`) is readable only by the `apinizer` user
(permissions `400`, owner `apinizer:apinizer`).

The on-disk encryption uses **AES-256** (`PBEWITHHMACSHA512ANDAES_256`)
with a random IV and salt per value.

**Encrypt the standard fields in `application.env` automatically (recommended):**

Run the script with no arguments. It opens `conf/application.env`,
wraps the values of the three standard fields with `ENC(...)` in place,
and creates a timestamped backup beside the file.

```bash
sudo -u apinizer /opt/apinizer-portal/bin/apiportal-encrypt.sh
```

Expected output:

```
[INFO] Backup: /opt/apinizer-portal/conf/application.env.backup-20260515-100214
[OK]   API_PORTAL_ID encrypted.
[OK]   API_PORTAL_MANAGEMENT_API_BASE_URL encrypted.
[OK]   API_PORTAL_MANAGEMENT_API_KEY encrypted.
[DONE] Encrypted: 3, already encrypted: 0, empty: 0
```

Fields covered by the batch run:

- `API_PORTAL_ID`
- `API_PORTAL_MANAGEMENT_API_BASE_URL`
- `API_PORTAL_MANAGEMENT_API_KEY`

The script is **idempotent** — values that are already wrapped in
`ENC(...)` or are empty are skipped; only plain values are encrypted.
Re-running the script after rotating credentials in plain form will
re-encrypt only the changed ones.

<Warning>
**Master key handling**

- Back up `conf/master.key` to a secure location (HSM, Vault, sealed
  envelope). If lost, all `ENC(...)` values must be re-encrypted with
  a freshly generated key.
- Never commit `master.key` to version control or place it under
  `application.env`.
- For key rotation: generate a new key, re-encrypt all `ENC(...)`
  values with `apiportal-encrypt.sh`, replace them in
  `application.env`, then restart the service.
</Warning>

## 6. Start the Service

```bash
sudo systemctl start apinizer-apiportal
sudo systemctl status apinizer-apiportal
```

Follow the live log:

```bash
sudo journalctl -u apinizer-apiportal -f
```

Once the application is up, the portal is reachable at:

```
http://<server-ip>:8080
```

## 7. Verify the Installation

```bash
curl -fsS http://localhost:8080/apiportal/management/health
# Expected: {"status":"UP"}
```

Check that the process is running under the dedicated user:

```bash
ps -ef | grep apinizer-apiportal | grep -v grep
# Should show user 'apinizer'
```

## Operations

### Service Control

```bash
sudo systemctl start apinizer-apiportal    # Start
sudo systemctl stop apinizer-apiportal     # Graceful stop
sudo systemctl restart apinizer-apiportal  # Restart
sudo systemctl status apinizer-apiportal   # Status
sudo systemctl enable apinizer-apiportal   # Enable on boot (default)
sudo systemctl disable apinizer-apiportal  # Disable on boot
```

### Logs

| Source          | Location                                                |
|-----------------|---------------------------------------------------------|
| systemd journal | `journalctl -u apinizer-apiportal -f`                   |
| stdout/stderr   | `/opt/apinizer-portal/logs/apinizer-apiportal.out`      |
| Logback file    | `/opt/apinizer-portal/logs/apinizer-apiportal.log`      |
| PID file        | `/opt/apinizer-portal/logs/apinizer-apiportal.pid`      |

### Manual Start (without systemd)

```bash
sudo -u apinizer /opt/apinizer-portal/bin/apiportal-start.sh              # background
sudo -u apinizer /opt/apinizer-portal/bin/apiportal-start.sh --foreground # foreground (debug)
sudo -u apinizer /opt/apinizer-portal/bin/apiportal-stop.sh               # graceful stop
```

### Memory Tuning

The start script auto-detects the available memory (cgroup limit or
host RAM) and sizes the JVM heap, direct memory, metaspace, and GC
parameters accordingly. To override these defaults, set `JAVA_OPTS`
in `application.env`:

```env
JAVA_OPTS=-Xms512m -Xmx2g -XX:+UseG1GC
jvmOverride=true
```

When `jvmOverride=true`, your `JAVA_OPTS` take precedence over the
auto-tier defaults.

## Security Model Summary

| Item                          | Permissions       | Owner                |
|-------------------------------|-------------------|----------------------|
| `conf/application.env`        | `600`             | `apinizer:apinizer`  |
| `conf/master.key`             | `400`             | `apinizer:apinizer`  |
| `bin/*.sh`                    | `700`             | `apinizer:apinizer`  |
| Service runs as               | `apinizer` (no login shell) |              |
| systemd hardening             | `NoNewPrivileges`, `ProtectSystem=strict`, `ProtectHome=true` | |

## Troubleshooting

**`[ERROR] application.env contains ENC(...) values but master key file is missing`**

The configuration references encrypted values but `conf/master.key` is
absent. Restore the key from your backup or re-run the installer to
generate a new one (note: a new key cannot decrypt values produced by
the previous key — re-encrypt them).

**`[WARN] /opt/apinizer-portal/conf/application.env permissions are 644 — should be 600 or 640`**

Run `sudo chmod 600 /opt/apinizer-portal/conf/application.env`.

**`Apinizer API Portal is already running (PID=...)`**

Either an instance is genuinely running, or a stale PID file remains.
Run `sudo systemctl stop apinizer-apiportal` and try again. If the PID
file persists with no matching process, the start script removes it
automatically.

**Port 8080 in use**

Change `SERVER_PORT` in `application.env` and restart the service.
