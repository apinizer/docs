This page covers installation of Apinizer's core modules on Linux / VM as standalone packages in a single place. Each module ships with its own embedded **OpenJDK 25** — no host Java required. Sensitive values (MongoDB connection URI etc.) are encrypted in place via a **Jasypt master key**. **Order matters**: first Manager (registration + Environment), then Cache, then Worker. Integration is only needed if you use task-flow scheduler. **API Portal has its own page** — install it after the Manager.

<Tabs>
  <Tab title="Overview">
    Apinizer ships as self-contained Linux packages with bundled OpenJDK 25 (Eclipse Temurin). No need to install Java on the host. The packages are designed for regulated environments (banking, government, defense) and systems where container runtimes are restricted.

    <Info>
    Installation runs through sudo — **no interactive root login is required**. This is compatible with hardened environments where direct root SSH is disabled (common in finance and regulated sectors).
    </Info>

    ## Common Package Layout

    Every module follows the same layout:

    ```
    apinizer-<module>-<version>/
    ├── bin/
    │   ├── <module>-start.sh       # Tier-aware JVM tuning + env load
    │   ├── <module>-stop.sh
    │   ├── <module>-install.sh     # sudo-friendly installer
    │   ├── <module>-encrypt.sh     # Jasypt batch + single-value
    │   └── apinizer-<module>.service  # systemd unit
    ├── conf/
    │   ├── application.env        # Plaintext or ENC(...) values
    │   └── master.key             # Generated by installer (chmod 400)
    ├── lib/
    │   ├── apinizer-<module>-<version>.jar   # Spring Boot fat jar
    │   └── jasypt/jasypt-*.jar              # For the encrypt CLI
    ├── runtime/                   # Bundled Temurin JDK 25
    └── logs/                      # Created at runtime
    ```

    ## Recommended Install Order

    1. **MongoDB** (must be installed first — see [MongoDB Installation](/en/setup/mongodb/ubuntu-mongodb))
    2. **API Manager** → connects to MongoDB, runs Mongock migrations
    3. **Worker** → fetches its Environment from Manager
    4. **Cache** → Hazelcast cluster (if Worker uses quota/OIDC/CB)
    5. **Integration** *(optional)* → Quartz scheduler, registers with Manager
    6. **API Portal** → connects to the Manager REST API

    ## Jasypt Master Key — Encrypted Configuration

    The biggest difference from Docker: **encryption-at-rest support**.

    - `<module>-install.sh` generates a random master key at `conf/master.key` (chmod 400, owned by `apinizer`)
    - `<module>-encrypt.sh` wraps the standard fields in place as `ENC(...)`
    - `<module>-start.sh` decrypts `ENC(...)` values at boot and passes plaintext to the JVM
    - The JVM never sees ciphertext; the master key never leaves the host filesystem

    Algorithm: `PBEWITHHMACSHA512ANDAES_256` (random IV/salt per value).

    <Warning>
    Back up `conf/master.key` to your secrets vault. **If the master key is lost, all `ENC(...)` values become unrecoverable**.
    </Warning>

    ## systemd Hardening

    All module service units enable the following hardening flags:

    ```ini
    NoNewPrivileges=true
    PrivateTmp=true              # MANDATORY — Undertow uses /tmp
    ProtectSystem=strict         # Whole fs read-only
    ProtectHome=true
    ReadWritePaths=/opt/apinizer-<module>/logs   # Only logs/ is writable
    ```

    <Warning>
    `PrivateTmp=true` is required alongside `ProtectSystem=strict` — without it Undertow cannot create `/tmp/undertow-docbase.*` and the service crashes.
    </Warning>

    ## sudo Delegation

    Sudoers example so a customer sysadmin can install without a root login:

    ```
    apinizer-admin ALL=(root) NOPASSWD: /opt/apinizer-<module>/bin/<module>-install.sh
    ```

  <Tab title="1. API Manager">
    This guide walks through installing the Apinizer API Manager standalone
    package on a Linux server. The package is fully self-contained — Java is
    **not** required on the host. Sensitive values (MongoDB connection URI
    and database name) are encrypted at rest with a randomly generated
    Jasypt master key.

    <Info>
    The installer needs root privileges via `sudo`. **No interactive root
    login is required**, which makes it compatible with hardened
    environments where direct root SSH is disabled (common in finance and
    regulated sectors).
    </Info>

    ## Requirements

    - Linux x86-64 (glibc) — Ubuntu 20.04+, RHEL/Rocky/Alma 8+, Debian 11+,
      Oracle Linux 8+
    - An operator with `sudo` privilege (no root login required)
    - `openssl`, `systemd`, `tar`, `curl` (typically pre-installed)
    - A reachable Apinizer Manager-managed MongoDB instance (replica set
      recommended for production)

    ## 1. Download

    Download the latest tarball from `packages.apinizer.com`:

    ```bash
    VERSION=2026.04.2
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/apimanager/${VERSION}/apinizer-apimanager-${VERSION}-linux-x64.tar.gz"
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/apimanager/${VERSION}/checksums.sha256"
    sha256sum -c checksums.sha256
    ```

    ## 2. Extract

    ```bash
    sudo mkdir -p /opt
    sudo tar -xzf apinizer-apimanager-${VERSION}-linux-x64.tar.gz -C /opt
    sudo mv /opt/apinizer-apimanager-${VERSION} /opt/apinizer-manager
    ```

    The package layout is:

    ```
    /opt/apinizer-manager/
    ├── bin/        # start / stop / install / encrypt scripts and systemd unit
    ├── conf/       # application.env (master.key generated by installer)
    ├── lib/        # Spring Boot fat jar + lib/jasypt/ helper jar
    ├── runtime/    # Bundled OpenJDK 25 (Temurin)
    └── logs/       # Runtime logs (created on first start)
    ```

    ## 3. Configure

    Edit `conf/application.env` and provide the **required** parameters:

    ```bash
    sudo vi /opt/apinizer-manager/conf/application.env
    ```

    | Key | Description |
    | --- | --- |
    | `SPRING_DATA_MONGODB_URI` | MongoDB connection string (e.g. `mongodb://user:pass@host:25080/?replicaSet=rs0&authSource=admin`) |
    | `SPRING_DATA_MONGODB_DATABASE` | Database name (e.g. `apinizer`) |

    Optional sections cover SSL, server port, JVM tuning, and logging path.

    <Warning>
    Plaintext values are accepted at this point — do not start the service
    yet. The next step encrypts them in place.
    </Warning>

    ## 4. Run the installer

    The installer creates a dedicated `apinizer` system user, sets file
    permissions, generates a random Jasypt master key in
    `conf/master.key`, and registers the systemd service.

    ```bash
    sudo /opt/apinizer-manager/bin/apimanager-install.sh
    ```

    To delegate this to a non-root operator without granting full root, add
    a sudoers entry:

    ```
    apinizer-admin ALL=(root) NOPASSWD: /opt/apinizer-manager/bin/apimanager-install.sh
    ```

    After install, the master key is owned by `apinizer:apinizer` and is
    mode `400` — only the service user can read it.

    ## 5. Encrypt sensitive values

    Run the encrypt helper with no arguments to encrypt the standard
    fields in `conf/application.env` in place. Already-encrypted (or empty)
    values are skipped, and a timestamped backup is written next to the
    file.

    ```bash
    sudo -u apinizer /opt/apinizer-manager/bin/apimanager-encrypt.sh
    ```

    The standard fields it encrypts are:

    - `SPRING_DATA_MONGODB_URI`
    - `SPRING_DATA_MONGODB_DATABASE`

    After it runs, the env file looks like this (values quoted so bash
    `source` does not choke on the parentheses):

    ```
    SPRING_DATA_MONGODB_URI="ENC(eXq1J9z...kTu4=)"
    SPRING_DATA_MONGODB_DATABASE="ENC(zMv3K7p...lQw9=)"
    ```

    ## 6. Start the service

    ```bash
    sudo systemctl start apinizer-apimanager
    sudo systemctl status apinizer-apimanager
    sudo journalctl -u apinizer-apimanager -f
    ```

    The service is enabled at install time, so it will also start
    automatically after a reboot.

    ## 7. Verify

    Confirm the API Manager is responding on the configured port (default `8080`):

    ```bash
    curl -fsS http://127.0.0.1:8080/management/health
    ```

    Expected: a `200 OK` response with health details (visible to
    authenticated admins; anonymous responses contain only the status).

    ## Security model

    - `apinizer` is a system account with `/sbin/nologin` — no interactive shell.
    - `conf/master.key` is mode `400`, owned by `apinizer`. Only that user
      (and root) can read it.
    - `conf/application.env` is mode `600`. ENC(...) values are decrypted
      by `bin/apimanager-start.sh` at boot (using the bundled
      `lib/jasypt/jasypt-*.jar` + the master key) and exported to the JVM
      as plain values. The JVM never sees the ciphertext; the master key
      never leaves the host filesystem.
    - The systemd unit applies hardening flags: `NoNewPrivileges`,
      `PrivateTmp`, `ProtectSystem=strict`, `ProtectHome`,
      `ReadWritePaths=…/logs`. Application can only write under `logs/`.

    <Warning>
    Back up `conf/master.key` to your secrets vault. **If the master key is
    lost, all ENC(...) values are unrecoverable** and must be re-entered
    in plaintext, then re-encrypted with a new key.
    </Warning>

    ## Troubleshooting

    ### Service exits immediately with "is not set"

    `SPRING_DATA_MONGODB_URI` or `SPRING_DATA_MONGODB_DATABASE` is empty
    in `conf/application.env`. Edit the file and re-run encryption.

    ### `Read-only file system` on `/tmp`

    The systemd unit hardening (`ProtectSystem=strict`) gives the service
    an isolated `/tmp` via `PrivateTmp=true`. If you replaced the unit and
    removed `PrivateTmp`, Undertow cannot create `/tmp/undertow-docbase.*`.
    Re-add `PrivateTmp=true`, then `daemon-reload` and `restart`.

    ### Encryption fails with "Master key file not found"

    `apimanager-install.sh` was never run, or the installer was run but
    did not generate a key. Re-run:

    ```bash
    sudo /opt/apinizer-manager/bin/apimanager-install.sh
    ```

    ### MongoDB connection times out

    - Check that the MongoDB hosts in `SPRING_DATA_MONGODB_URI` are
      reachable from the server (`nc -zv host 25080`).
    - Verify replica set name and `authSource` are correct.
    - If TLS is required, append `&tls=true` (and any required CA file
      options) to the URI.

  <Tab title="3. API Cache">
    This guide walks through installing the Apinizer Cache (the Hazelcast
    data grid that backs quotas, OIDC sessions, circuit-breaker state and
    client-flow banners) standalone package on a Linux server. The package
    is fully self-contained — Java is **not** required on the host.
    Sensitive values (MongoDB connection URI and database name) are
    encrypted at rest with a randomly generated Jasypt master key.

    <Info>
    The installer needs root privileges via `sudo`. **No interactive root
    login is required**, which makes it compatible with hardened
    environments where direct root SSH is disabled (common in finance and
    regulated sectors).
    </Info>

    ## Requirements

    - Linux x86-64 (glibc) — Ubuntu 20.04+, RHEL/Rocky/Alma 8+, Debian 11+,
      Oracle Linux 8+
    - An operator with `sudo` privilege (no root login required)
    - `openssl`, `systemd`, `tar`, `curl` (typically pre-installed)
    - A reachable Apinizer Manager-managed MongoDB instance (replica set
      recommended for production — the Cache writes quota/usage snapshots
      to MongoDB via Hazelcast MapStore)
    - Open ports between Cache nodes and Workers:
      - `5701/tcp` — Hazelcast wire protocol (member-to-member)
      - `8090/tcp` — Spring Boot HTTP (Worker → Cache REST calls)

    ## 1. Download

    Download the latest tarball from `packages.apinizer.com`:

    ```bash
    VERSION=2026.04.5
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/cache/${VERSION}/apinizer-cache-${VERSION}-linux-x64.tar.gz"
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/cache/${VERSION}/checksums.sha256"
    sha256sum -c checksums.sha256
    ```

    ## 2. Extract

    ```bash
    sudo mkdir -p /opt
    sudo tar -xzf apinizer-cache-${VERSION}-linux-x64.tar.gz -C /opt
    sudo mv /opt/apinizer-cache-${VERSION} /opt/apinizer-cache
    ```

    The package layout is:

    ```
    /opt/apinizer-cache/
    ├── bin/        # start / stop / install / encrypt scripts and systemd unit
    ├── conf/       # apinizer-cache.env (master.key generated by installer)
    ├── lib/        # Cache Spring Boot fat jar + lib/jasypt/ helper jar
    ├── runtime/    # Bundled OpenJDK 25 (Temurin)
    └── logs/       # Runtime logs (created on first start)
    ```

    ## 3. Configure

    Edit `conf/apinizer-cache.env` and provide the **required** parameters:

    ```bash
    sudo vi /opt/apinizer-cache/conf/apinizer-cache.env
    ```

    | Key | Description |
    | --- | --- |
    | `SPRING_DATA_MONGODB_URI` | MongoDB connection string (e.g. `mongodb://user:pass@host:25080/?replicaSet=rs0&authSource=admin`) |
    | `SPRING_DATA_MONGODB_DATABASE` | Database name (e.g. `apinizer`) |
    | `CACHE_QUOTA_TIMEZONE` | Quota window timezone offset (e.g. `+03:00`). Must match the Worker and Manager values or quota rollovers will drift. |

    Standalone mode is **auto-detected** — when the Kubernetes service-account
    namespace file (`/var/run/secrets/kubernetes.io/serviceaccount/namespace`)
    is absent, the Cache switches Hazelcast discovery from the Kubernetes
    plug-in to TCP/IP. No extra flag is required.

    <Info>
    The Hazelcast logical cluster name (`HAZELCAST_CLUSTER_NAME`) ships with
    a stable default (`apnz-hz-cluster`). Leave it as-is unless you operate
    **multiple independent Apinizer installs on the same LAN**, in which
    case give each install a unique name. A mismatch between Cache nodes
    and Workers silently splits the grid — HTTP keeps working but quota
    counters fail to converge.
    </Info>

    ### Single-node vs multi-node

    For a single-node deployment no further settings are needed. The
    default `APINIZER_CACHE_CLUSTER_MEMBERS` (empty) is treated as
    `127.0.0.1` and a one-member cluster forms locally.

    For a **multi-node** cluster set the same comma-separated peer list on
    every node:

    ```bash
    APINIZER_CACHE_CLUSTER_MEMBERS=10.0.0.11,10.0.0.12,10.0.0.13
    ```

    <Tip>
    All members of the cluster must list **exactly the same set of IPs**.
    TCP/IP discovery in Hazelcast is symmetric — if node B does not list A,
    B will not accept A as a member even though A reaches B.
    </Tip>

    <Warning>
    Plaintext values are accepted at this point — do not start the service
    yet. The next step encrypts them in place.
    </Warning>

    ## 4. Encrypt credentials

    Run the install script first; it creates the `apinizer` user, sets
    permissions, generates `conf/master.key`, validates the env file, and
    registers the systemd unit:

    ```bash
    sudo /opt/apinizer-cache/bin/apicache-install.sh
    ```

    Then encrypt the MongoDB credentials in place:

    ```bash
    sudo -u apinizer /opt/apinizer-cache/bin/apicache-encrypt.sh
    ```

    This wraps `SPRING_DATA_MONGODB_URI` and `SPRING_DATA_MONGODB_DATABASE`
    with `ENC(...)` using the Jasypt master key. The JVM never sees the
    ciphertext — the start script decrypts each `ENC(...)` entry **before**
    launching Java.

    To encrypt a single arbitrary value:

    ```bash
    sudo -u apinizer /opt/apinizer-cache/bin/apicache-encrypt.sh '<plaintext>'
    ```

    <Warning>
    Back up `conf/master.key` to your secrets vault before changing anything
    under `conf/`. If the key is lost, every `ENC(...)` value becomes
    unrecoverable.
    </Warning>

    ## 5. Start the service

    ```bash
    sudo systemctl start apinizer-apicache
    sudo systemctl status apinizer-apicache
    sudo journalctl -u apinizer-apicache -f
    ```

    Expected startup log lines:

    ```
    Cache running in STANDALONE mode (cluster: apnz-hz-cluster)
    Standalone Hazelcast cluster members: 10.0.0.11,10.0.0.12,10.0.0.13
    Hazelcast thread pools configured successfully
    ...
    Members {size:3, ver:3} [
        Member [10.0.0.11]:5701 - ...
        Member [10.0.0.12]:5701 - ...
        Member [10.0.0.13]:5701 - ...
    ]
    ```

    If only one member is listed in a multi-node setup, check firewall
    rules on `5701/tcp` and that every node lists the same peers.

    ## 6. Wire Workers to the Cache

    Each Worker (`apinizer-worker`) must use the same Hazelcast cluster
    name and reach all Cache nodes on `5701/tcp`. The Worker reads the
    cluster name from its own configuration; no Worker-side change is
    needed beyond keeping the names in sync.

    ## 7. Manual control (without systemd)

    ```bash
    sudo -u apinizer /opt/apinizer-cache/bin/apicache-start.sh           # background
    sudo -u apinizer /opt/apinizer-cache/bin/apicache-start.sh -f        # foreground
    sudo -u apinizer /opt/apinizer-cache/bin/apicache-stop.sh
    ```

    ## 8. Tuning (optional)

    Hazelcast performance knobs, MapStore write-behind sizes, split-brain
    protection and CP Subsystem options are all documented inline in
    `apinizer-cache.env`. The shipped defaults match the values used in
    the Kubernetes deployment, so re-tune only after a sustained load test
    on the target hardware.
  </Tab>
  <Tab title="4. API Integration (Optional)">
    This guide walks through installing the Apinizer Integration tier — the
    Quartz scheduler that runs task flows registered by the Manager — as a
    standalone Linux service. The package is fully self-contained — Java is
    **not** required on the host. Sensitive values (MongoDB connection URI
    and database name) are encrypted at rest with a randomly generated
    Jasypt master key.

    <Info>
    The installer needs root privileges via `sudo`. **No interactive root
    login is required**, which makes it compatible with hardened
    environments where direct root SSH is disabled (common in finance and
    regulated sectors).
    </Info>

    ## Requirements

    - Linux x86-64 (glibc) — Ubuntu 20.04+, RHEL/Rocky/Alma 8+, Debian 11+,
      Oracle Linux 8+
    - An operator with `sudo` privilege (no root login required)
    - `openssl`, `systemd`, `tar`, `curl` (typically pre-installed)
    - A reachable Apinizer Manager-managed MongoDB instance (the Integration
      reads task-flow / connector definitions from the same database the
      Manager writes to, and stores its Quartz job state in
      `schedule_integration_*` collections inside it)
    - Open ports:
      - `8092/tcp` — Spring Boot HTTP (Manager → Integration REST calls).
        The Integration does NOT need to be reachable from Workers or
        end-users; lock it down to the Manager's IP/CIDR.

    ## 1. Download

    Download the latest tarball from `packages.apinizer.com`:

    ```bash
    VERSION=2026.04.5
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/integration/${VERSION}/apinizer-integration-${VERSION}-linux-x64.tar.gz"
    curl -fSLO "https://packages.apinizer.com/apinizer-packages/integration/${VERSION}/checksums.sha256"
    sha256sum -c checksums.sha256
    ```

    ## 2. Extract

    ```bash
    sudo mkdir -p /opt
    sudo tar -xzf apinizer-integration-${VERSION}-linux-x64.tar.gz -C /opt
    sudo mv /opt/apinizer-integration-${VERSION} /opt/apinizer-integration
    ```

    The package layout is:

    ```
    /opt/apinizer-integration/
    ├── bin/        # start / stop / install / encrypt scripts and systemd unit
    ├── conf/       # apinizer-integration.env (master.key generated by installer)
    ├── lib/        # Integration Spring Boot fat jar + lib/jasypt/ helper jar
    ├── runtime/    # Bundled OpenJDK 25 (Temurin)
    └── logs/       # Runtime logs (created on first start)
    ```

    ## 3. Configure

    Edit `conf/apinizer-integration.env` and provide the **required** parameters:

    ```bash
    sudo vi /opt/apinizer-integration/conf/apinizer-integration.env
    ```

    | Key | Description |
    | --- | --- |
    | `SPRING_DATA_MONGODB_URI` | MongoDB connection string (e.g. `mongodb://user:pass@host:25080/?replicaSet=rs0&authSource=admin`) — same instance the Manager uses |
    | `SPRING_DATA_MONGODB_DATABASE` | Database name (e.g. `apinizer`) — must match the Manager database |
    | `INTEGRATION_TIMEZONE` | IANA zone id (e.g. `Europe/Istanbul`). Must match the Manager and Worker timezone, otherwise cron jobs fire at the wrong wall-clock time. |

    <Warning>
    Plaintext values are accepted at this point — do not start the service
    yet. The next step encrypts them in place.
    </Warning>

    ```
    Integration timezone set to: Europe/Istanbul
    Initializing Custom MongoDB Job Store
    Quartz Scheduler Prod Environment Setup started
    MongoDB connection established for Quartz Job Store
    Quartz MongoDB Statistics - Jobs: ..., Triggers: ..., Calendars: ..., Locks: ...
    ```

    ## 6. Multi-node / HA

    Multiple Integration nodes can share the **same MongoDB database** —
    they form a clustered Quartz scheduler automatically. Each instance:

    - Auto-generates its `org.quartz.scheduler.instanceId` (no manual
      configuration needed).
    - Locks triggers via the `schedule_integration_locks` collection so the
      same job never fires on two nodes at once.
    - Recovers misfired jobs on startup (triggers stuck in
      `ACQUIRED`/`EXECUTING` state are reset to `WAITING`).

    No additional standalone setting is needed — just point every node at
    the same MongoDB URI + database.

    <Tip>
    Stand up Integration nodes in pairs at minimum if you need
    fail-over for time-critical scheduled flows. With one node, a host
    restart blocks scheduled runs until the JVM is back up.
    </Tip>

    ## 7. Wire the Manager to the Integration

    In the Manager UI (`Admin → Integration Servers`) add this host's
    hostname/IP and the HTTP port (`8092`). The Manager pushes job
    register/unregister and cache-refresh events via REST; no other
    configuration is required.

    ```bash
    sudo -u apinizer /opt/apinizer-integration/bin/apiintegration-start.sh -f        # foreground
    sudo -u apinizer /opt/apinizer-integration/bin/apiintegration-stop.sh
    ```

    ## 9. Tuning (optional)

    JVM heap percentage, server port, log directory and log level are all
    documented inline in `apinizer-integration.env`. The shipped defaults
    match the values used in the Kubernetes deployment, so re-tune only
    after a sustained load test on the target hardware.

    <Info>
    If you run task flows that commonly take minutes to finish, raise
    `TimeoutStopSec=` in the systemd unit (defaults to 120 s). Quartz is
    configured with `waitForJobsToCompleteOnShutdown=true`, so a SIGTERM
    triggers a clean drain — the timeout caps how long systemd waits before
    escalating to SIGKILL.
    </Info>
