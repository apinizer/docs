Apinizer modules are published on Docker Hub as official images under `apinizercloud/<module>`. Each image is built on Eclipse Temurin 25 (Linux x86-64, glibc). The container entrypoint reads the memory limit at boot and automatically selects tier-aware JVM heap and metaspace profiles — no manual JVM tuning is needed for most deployments.

<Info>
**docker-compose** is recommended for production — it manages dependencies, networking, and volumes declaratively in one file. Single-container `docker run` is suitable for test or proof-of-concept deployments.
</Info>

## Installation Flow

Follow this sequence to install Apinizer modules on Docker:

1. **MongoDB** — Required. Must be pre-configured and reachable. See [MongoDB Installation](/en/setup/mongodb).
2. **API Manager** — The management console. Install this first.
3. **Gateway (Worker)** — The API Gateway runtime. Install after Manager and register as an environment.
4. **Optional:** Cache (Hazelcast), Integration (Quartz), API Portal

## General Requirements

Common to all Apinizer Docker deployments:

- **Docker Engine 24+** and **docker-compose v2+**
- **CPU**: At least 2 cores (per module)
- **RAM**: At least 2 GB (per module); 4 GB+ recommended for production
- **Disk**: ~1 GB per module image + volumes for logs
- **Network**: TCP connectivity to MongoDB and between modules
- **MongoDB**: Pre-installed and reachable (replica set recommended for production)
- **Java**: Not required — OpenJDK 25 is bundled in every image

<Warning>
MongoDB is mandatory for all modules. Installation is out of scope for this document. You need a reachable MongoDB instance before launching Apinizer.
</Warning>

## API Manager Installation

The Apinizer Management Console (**API Manager**) is the control plane — REST API, admin UI, proxy lifecycle, environment management, and user access control.

### 1. Pull the Image

```bash
VERSION=2026.04.5
docker pull apinizercloud/apimanager:${VERSION}
```

See [Docker image tag convention](/en/setup/docker/overview#tag-convention) for version details.

### 2. Configure

Required environment variables (the container exits at startup if they are missing):

| Variable | Description |
| --- | --- |
| `SPRING_DATA_MONGODB_URI` | MongoDB connection string (e.g., `mongodb://user:pass@mongo:25080/?replicaSet=rs0&authSource=admin`) |
| `SPRING_DATA_MONGODB_DATABASE` | Database name (e.g., `apinizer`) |
| `SPRING_PROFILES_ACTIVE` | `prod` (default — keep on production hosts) |

Optional:

| Variable | Default | Description |
| --- | --- | --- |
| `SERVER_PORT` | `8080` | HTTP port inside the container |
| `JAVA_OPTS` | empty | Extra JVM flags (merged with tier-aware defaults) |
| `jvmOverride` | `false` | When `true`, `JAVA_OPTS` takes precedence over tier-aware defaults |
| `JHIPSTER_SLEEP` | `0` | Pre-start delay in seconds (useful when MongoDB starts in the same compose) |

<Tip>
Sensitive values (MongoDB URI, credentials) are passed as plaintext environment variables or via `env_file` entries in docker-compose. Use Docker Secrets (Compose / Swarm) or Kubernetes Secrets to inject them securely in production.
</Tip>

### 3. Start the Container (docker-compose recommended)

```yaml
services:
  apimanager:
    image: apinizercloud/apimanager:2026.04.5
    container_name: apinizer-apimanager
    restart: unless-stopped
    mem_limit: 4g
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: prod
      SPRING_DATA_MONGODB_DATABASE: apinizer
      JAVA_OPTS: "-XX:+ExitOnOutOfMemoryError"
    env_file:
      - ./secrets/apimanager.env       # contains SPRING_DATA_MONGODB_URI=…
    volumes:
      - apimanager-logs:/app/logs
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:8080/management/health | grep -q '\"status\":\"UP\"'"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 90s

volumes:
  apimanager-logs:
```

Start:

```bash
docker compose up -d
docker compose logs -f apimanager
```

Or use `docker run` for simple deployments:

```bash
docker run -d \
  --name apinizer-apimanager \
  -p 8080:8080 \
  --memory=4g \
  -e SPRING_DATA_MONGODB_URI='mongodb://user:pass@mongo:25080/?authSource=admin' \
  -e SPRING_DATA_MONGODB_DATABASE='apinizer' \
  -e SPRING_PROFILES_ACTIVE=prod \
  -v apinizer-apimanager-logs:/app/logs \
  --restart unless-stopped \
  apinizercloud/apimanager:${VERSION}
```

<Tip>
The `--memory=4g` flag caps the container memory; the JVM tier-tuner also reads this and sizes the heap accordingly. Without it, the JVM sees the full host memory.
</Tip>

### 4. Verify

```bash
curl -fsS http://127.0.0.1:8080/management/health
```

Expected: a `200 OK` response with health details. The first start runs Mongock migrations and may take 30–90 seconds.

### 5. Operations

<CardGroup cols={2}>
  <Card title="Status">docker compose ps</Card>
  <Card title="Restart">docker compose restart apimanager</Card>
  <Card title="Stop">docker compose stop apimanager</Card>
  <Card title="Logs">docker compose logs -f apimanager</Card>
</CardGroup>

### 6. Networking

- **Inbound**: `8080/tcp` must be reachable by Worker, Portal, and admin users.
- **Outbound**: Connects to MongoDB and configured backend services (LDAP, OIDC, SMTP). No outbound internet required in steady state.
- For multi-container deployments, place all Apinizer services on the same user-defined Docker network and reference them by service name.

### 7. Resources

| Workload | `mem_limit` | CPU |
| --- | --- | --- |
| Smoke test / single tenant | 2 GB | 2 vCPU |
| Production (≤200 active proxies) | 4 GB | 4 vCPU |
| Large install / many concurrent admins | 6–8 GB | 4–8 vCPU |

<Note>
For API Manager troubleshooting, see [Troubleshooting](/en/setup/troubleshooting).
</Note>

## Gateway (Worker) Installation and Definition

The Gateway (**Worker**) is Apinizer's API Gateway runtime — the entry point for HTTP/HTTPS/gRPC/WebSocket requests, the execution engine for deployed proxy policies, and the source of traffic logs.

### 1. Pull the Image

```bash
VERSION=2026.04.5
docker pull apinizercloud/worker:${VERSION}
```

<Info>
The Worker image is built on **Ubuntu noble (glibc)** — not Alpine — because the `brotli4j` native compression library has no musl build.
</Info>

### 2. Configure

Required environment variables:

| Variable | Description |
| --- | --- |
| `SPRING_DATA_MONGODB_URI` | MongoDB connection string (same instance as Manager) |
| `SPRING_DATA_MONGODB_DATABASE` | MongoDB database name (same as Manager) |
| `APINIZER_ENVIRONMENT_NAME` | Environment name. **Must exactly match** an Environment in the Manager UI (case-sensitive). |
| `WORKER_TIMEZONE` | Timezone offset for log timestamps (e.g., `+03:00`) |

Recommended:

| Variable | Default | Description |
| --- | --- | --- |
| `JAVA_OPTS` | empty | Production: `"-server -XX:MaxRAMPercentage=75.0 -Dhttp.maxConnections=4096 -Dlog4j.formatMsgNoLookups=true"` |

Optional (Undertow/gateway tuning):

| Variable | Default | Description |
| --- | --- | --- |
| `tuneWorkerThreads` | `1024` | XNIO worker thread pool |
| `tuneWorkerMaxThreads` | `4096` | Hard cap on worker pool |
| `tuneIoThreads` | `4` | XNIO I/O selectors (match vCPU count) |
| `tuneBufferSize` | `16384` | Read/write buffer size in bytes |
| `tuneDirectBuffers` | `true` | Off-heap buffers (less GC pressure) |
| `tuneBacklog` | `10000` | Kernel TCP listen backlog |

### 3. Start the Container

```yaml
services:
  worker:
    image: apinizercloud/worker:2026.04.5
    container_name: apinizer-worker
    restart: unless-stopped
    mem_limit: 8g
    cpus: 4
    ports:
      - "8091:8091"
      - "8443:8443"
    environment:
      SPRING_DATA_MONGODB_DATABASE: apinizer
      APINIZER_ENVIRONMENT_NAME: prod-dc1
      WORKER_TIMEZONE: "+03:00"
      JAVA_OPTS: "-server -XX:MaxRAMPercentage=75.0 -Dhttp.maxConnections=4096 -Dlog4j.formatMsgNoLookups=true"
      tuneWorkerThreads: "1024"
      tuneIoThreads: "4"
    env_file:
      - ./secrets/worker.env
    ulimits:
      nofile:
        soft: 1048576
        hard: 1048576
    volumes:
      - worker-logs:/app/logs
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:8091/server-status | grep -q OK || exit 1"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 60s

volumes:
  worker-logs:
```

Start:

```bash
docker compose up -d
docker compose logs -f worker
```

Or use `docker run`:

```bash
docker run -d \
  --name apinizer-worker \
  -p 8091:8091 \
  -p 8443:8443 \
  --memory=8g \
  --cpus=4 \
  -e SPRING_DATA_MONGODB_URI='mongodb://user:pass@mongo:25080/?authSource=admin' \
  -e SPRING_DATA_MONGODB_DATABASE='apinizer' \
  -e APINIZER_ENVIRONMENT_NAME='prod-dc1' \
  -e WORKER_TIMEZONE='+03:00' \
  -e JAVA_OPTS='-server -XX:MaxRAMPercentage=75.0 -Dhttp.maxConnections=4096 -Dlog4j.formatMsgNoLookups=true' \
  -v apinizer-worker-logs:/app/logs \
  --restart unless-stopped \
  --ulimit nofile=1048576:1048576 \
  apinizercloud/worker:${VERSION}
```

<Tip>
The Worker is throughput-sensitive and benefits from the `--ulimit nofile=1048576:1048576` flag for high concurrent connection counts.
</Tip>

### 4. Verify

```bash
curl -fsS http://127.0.0.1:8091/server-status
```

Send a test request through a deployed proxy:

```bash
curl -fsS http://127.0.0.1:8091/echo -H 'X-Trace: 1'
```

Expected boot logs:

```
ApiGateway standalone=true (env: prod-dc1)
HTTP listener bound: 0.0.0.0:8091
HTTPS listener bound: 0.0.0.0:8443    (if HTTPS is enabled in the Environment)
Loaded N proxies for environment prod-dc1
```

### 5. Operations

<CardGroup cols={2}>
  <Card title="Status">docker compose ps</Card>
  <Card title="Logs">docker compose logs -f worker</Card>
  <Card title="Restart">docker compose restart worker</Card>
  <Card title="Stop (drain)">docker compose stop -t 60 worker</Card>
</CardGroup>

<Warning>
Use a generous stop timeout (`docker stop -t 60`) so in-flight HTTP/gRPC/WebSocket requests can drain gracefully. Without `-t`, Docker kills with SIGTERM after 10 seconds.
</Warning>

### 6. Networking

| Port | Purpose |
| --- | --- |
| `8091` | HTTP gateway listener |
| `8443` | HTTPS gateway listener (when enabled in the Environment) |

Outbound:
- MongoDB (proxy config + traffic log writes)
- Cache cluster on `5701/tcp` (Hazelcast) and `8090/tcp` (REST) — required if quotas/OIDC/circuit-breakers are used
- Each backend service of deployed proxies

### 7. Horizontal Scale (Multi-Worker)

Run the same image with the **same** `APINIZER_ENVIRONMENT_NAME` on each replica. Quota counters, OIDC sessions, and circuit-breaker state are shared through the **Apinizer Cache** (Hazelcast). Without a shared Cache, each Worker maintains its own local counters.

### 8. Resources

| Workload | `mem_limit` | `cpus` |
| --- | --- | --- |
| Smoke test / staging (&lt;200 rps) | 2 GB | 2 |
| Production (1–3 krps per Worker) | 4–6 GB | 4 |
| High throughput (5+ krps) | 8 GB | 4–8 |

Always load-test on target hardware before changing JVM flags.

<Note>
For Gateway troubleshooting, see [Troubleshooting](/en/setup/troubleshooting).

For additional configuration (gRPC, WebSocket, CORS, security, etc.), see [Gateway Settings](/en/setup/gateway-settings).
</Note>

## Defining the Gateway as an Environment in the Manager

After the Gateway container is running, register it in the Manager UI as an environment so you can deploy proxies to it.

1. Go to **Server Management → Gateway Runtimes → New**
2. Fill the registration form (see below)
3. Click **Create**

<Frame>
  <img
    src="/images/yonetici/envVm.png"
    alt="Gateway environment registration form (Standalone mode)"
  />
</Frame>

### Form Fields

| Field | Value |
|-------|-------|
| **Platform** | Standalone (Docker / VM / Bare metal) |
| **Environment Type** | Production / Test / Development |
| **Communication Protocol Type** | HTTP / HTTPS / gRPC / WebSocket (match what the Gateway will serve) |
| **Environment Name** | Must exactly match `APINIZER_ENVIRONMENT_NAME` (case-sensitive) |
| **Environment Key** | API URL prefix for this environment (e.g., `prod`) |
| **Access URL** | Gateway external access URL (e.g., `http://gateway.example.com:8091` or `https://gateway.example.com`) |

### Gateway Node List

When **Standalone** is selected, a **Gateway Node List** table appears. For each node (Gateway container):

| Field | Description |
|-------|-------------|
| **Node Name** | Identifier (e.g., `gateway-1`) |
| **Gateway Management API URL** | `http://<HOST_IP>:8091` (must be reachable from Manager) |
| **Cache Management API URL** | `http://<CACHE_HOST>:8090` (if a Cache is deployed) |

<Warning>
These management API URLs must be reachable from the Manager container. With Docker bridge networks, ensure routing or DNS is configured so the Manager can reach the Worker and Cache containers by hostname or IP.
</Warning>

### Publishing the Environment

Click **Create**. Apinizer does NOT deploy the Gateway container — it was already started above. The Manager only registers the connection metadata and attempts to verify connectivity.

<Info>
After publishing, you can deploy API Proxies to this Environment.
</Info>

## All-in-One docker-compose Example

If you want to bring up all Apinizer modules and an evaluation MongoDB on a single host with one `docker-compose.yml`, use the reference example below. Intended for:

- Evaluation / proof-of-concept
- Developer machines
- Internal demos

<Warning>
This compose is **not** production-ready as-is. For production, split MongoDB onto a dedicated replica set, replace plaintext secrets with Docker Secrets or a secret manager, terminate TLS at a reverse proxy, and size resources for your traffic.
</Warning>

### File Structure

```
apinizer-stack/
├── docker-compose.yml
├── .env                       # version + MongoDB creds
└── secrets/
    ├── apimanager.env
    ├── apiworker.env
    ├── apicache.env
    ├── apiintegration.env
    └── apiportal.env
```

### `.env`

```dotenv
VERSION=2026.04.5
MONGO_ROOT_USER=apinizer
MONGO_ROOT_PASSWORD=changeme-please
TIMEZONE_OFFSET=+03:00
TIMEZONE_IANA=Europe/Istanbul
```

### `docker-compose.yml`

```yaml
name: apinizer

services:
  # MongoDB (single-node for evaluation; use a replica set in prod)
  mongo:
    image: mongo:7
    container_name: apinizer-mongo
    restart: unless-stopped
    command: ["mongod", "--port", "25080"]
    environment:
      MONGO_INITDB_ROOT_USERNAME: ${MONGO_ROOT_USER}
      MONGO_INITDB_ROOT_PASSWORD: ${MONGO_ROOT_PASSWORD}
    volumes:
      - mongo-data:/data/db
    ports:
      - "25080:25080"
    healthcheck:
      test: ["CMD", "mongosh", "--port", "25080", "--quiet", "--eval", "db.adminCommand('ping').ok"]
      interval: 10s
      timeout: 5s
      retries: 10
      start_period: 20s

  # API Manager
  apimanager:
    image: apinizercloud/apimanager:${VERSION}
    container_name: apinizer-apimanager
    restart: unless-stopped
    depends_on:
      mongo:
        condition: service_healthy
    mem_limit: 4g
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: prod
      SPRING_DATA_MONGODB_DATABASE: apinizer
    env_file:
      - ./secrets/apimanager.env
    volumes:
      - apimanager-logs:/app/logs
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:8080/management/health | grep -q '\"status\":\"UP\"'"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 90s

  # Cache (Hazelcast)
  cache:
    image: apinizercloud/cache:${VERSION}
    container_name: apinizer-cache
    restart: unless-stopped
    depends_on:
      mongo:
        condition: service_healthy
    mem_limit: 4g
    ports:
      - "5701:5701"
      - "8090:8090"
    environment:
      SPRING_DATA_MONGODB_DATABASE: apinizer
      CACHE_QUOTA_TIMEZONE: "${TIMEZONE_OFFSET}"
      HAZELCAST_CLUSTER_NAME: apnz-hz-cluster
    env_file:
      - ./secrets/apicache.env
    volumes:
      - cache-logs:/app/logs

  # Worker (Gateway)
  worker:
    image: apinizercloud/worker:${VERSION}
    container_name: apinizer-worker
    restart: unless-stopped
    depends_on:
      apimanager:
        condition: service_healthy
      cache:
        condition: service_started
    mem_limit: 8g
    cpus: 4
    ports:
      - "8091:8091"
      - "8443:8443"
    environment:
      SPRING_DATA_MONGODB_DATABASE: apinizer
      APINIZER_ENVIRONMENT_NAME: prod-dc1
      WORKER_TIMEZONE: "${TIMEZONE_OFFSET}"
      JAVA_OPTS: "-server -XX:MaxRAMPercentage=75.0 -Dhttp.maxConnections=4096 -Dlog4j.formatMsgNoLookups=true"
    env_file:
      - ./secrets/apiworker.env
    ulimits:
      nofile:
        soft: 1048576
        hard: 1048576
    volumes:
      - worker-logs:/app/logs

  # Integration (Quartz)
  integration:
    image: apinizercloud/integration:${VERSION}
    container_name: apinizer-integration
    restart: unless-stopped
    depends_on:
      apimanager:
        condition: service_healthy
    mem_limit: 2g
    stop_grace_period: 120s
    ports:
      - "8092:8092"
    environment:
      SPRING_DATA_MONGODB_DATABASE: apinizer
      INTEGRATION_TIMEZONE: "${TIMEZONE_IANA}"
      SPRING_PROFILES_ACTIVE: prod
    env_file:
      - ./secrets/apiintegration.env
    volumes:
      - integration-logs:/app/logs

  # API Portal
  apiportal:
    image: apinizercloud/apiportal:${VERSION}
    container_name: apinizer-apiportal
    restart: unless-stopped
    depends_on:
      apimanager:
        condition: service_healthy
    mem_limit: 2g
    ports:
      - "8081:8080"
    environment:
      API_PORTAL_ID: portal-prod-01
      API_PORTAL_MANAGEMENT_API_BASE_URL: http://apimanager:8080
      SPRING_PROFILES_ACTIVE: prod
    env_file:
      - ./secrets/apiportal.env
    volumes:
      - apiportal-logs:/app/logs

volumes:
  mongo-data:
  apimanager-logs:
  cache-logs:
  worker-logs:
  integration-logs:
  apiportal-logs:
```

### Bring-Up Order

```bash
# 1. Start MongoDB and wait for it to be healthy
docker compose up -d mongo

# 2. Start the Manager — it runs the Mongock migrations
docker compose up -d apimanager
docker compose logs -f apimanager   # wait for "Started ApinizerManagerApp"

# 3. Create a Portal record + API key in the Manager UI
#    (http://localhost:8080 — default admin password is set on first start).
#    Put the API key into secrets/apiportal.env:
#    API_PORTAL_MANAGEMENT_API_KEY=<copied-key>

# 4. Create an Environment named "prod-dc1" in the Manager UI
#    (Server Management → Gateway Runtimes → New) so the Worker can bind to it.

# 5. Bring up the rest
docker compose up -d cache worker integration apiportal
```

<Tip>
The Portal and Worker depend on the Manager being bootstrapped (Portal ID issued, Environment created). The compose's `depends_on` only waits for the healthcheck — manual steps in the Manager UI must happen between step 2 and step 5.
</Tip>

### Operations

```bash
docker compose ps                # status
docker compose logs -f apimanager
docker compose restart worker
docker compose down              # stop everything (keeps volumes)
docker compose down -v           # stop and DELETE volumes (full reset)
```

## Optional Modules

<Tabs>
  <Tab title="Cache Installation">
    ## Apinizer Cache

    The Apinizer **Cache** is a Hazelcast data grid that backs quotas, OIDC sessions, circuit-breaker state, and client-flow banners. Install it after the Manager and before the Worker so Workers can join on first boot.

    ### 1. Pull the Image

    ```bash
    VERSION=2026.04.5
    docker pull apinizercloud/cache:${VERSION}
    ```

    ### 2. Configure

    Required:

    | Variable | Description |
    | --- | --- |
    | `SPRING_DATA_MONGODB_URI` | MongoDB connection string |
    | `SPRING_DATA_MONGODB_DATABASE` | MongoDB database (same as Manager/Worker) |
    | `CACHE_QUOTA_TIMEZONE` | Quota window timezone offset (e.g., `+03:00`). **Must match** Worker and Manager. |

    Recommended:

    | Variable | Default | Description |
    | --- | --- | --- |
    | `HAZELCAST_CLUSTER_NAME` | `apnz-hz-cluster` | Logical cluster name. Override only if running multiple independent Apinizer installs on the same LAN. **Mismatch with Worker causes silent split-brain.** |
    | `APINIZER_CACHE_CLUSTER_MEMBERS` | empty | Comma-separated peer list for multi-node TCP/IP discovery |
    | `SERVER_PORT` | `8090` | Spring Boot HTTP port |

    ### 3. Single-Node Cache

    ```bash
    docker run -d \
      --name apinizer-cache \
      -p 5701:5701 \
      -p 8090:8090 \
      --memory=4g \
      -e SPRING_DATA_MONGODB_URI='mongodb://user:pass@mongo:25080/?authSource=admin' \
      -e SPRING_DATA_MONGODB_DATABASE='apinizer' \
      -e CACHE_QUOTA_TIMEZONE='+03:00' \
      -v apinizer-cache-logs:/app/logs \
      --restart unless-stopped \
      apinizercloud/cache:${VERSION}
    ```

    Expected boot log:

    ```
    Cache running in STANDALONE mode (cluster: apnz-hz-cluster)
    Standalone Hazelcast cluster members: 127.0.0.1
    Members {size:1, ver:1} [ Member [127.0.0.1]:5701 - ... ]
    ```

    ### 4. Multi-Node Cache Cluster

    Each node must have the **same** peer list and **the same cluster name**.

    #### Compose (single-host, three replicas — for testing)

    ```yaml
    services:
      cache-1:
        image: apinizercloud/cache:2026.04.5
        container_name: apinizer-cache-1
        networks:
          apinizer:
            ipv4_address: 172.30.0.11
        environment: &cache-env
          SPRING_DATA_MONGODB_DATABASE: apinizer
          CACHE_QUOTA_TIMEZONE: "+03:00"
          HAZELCAST_CLUSTER_NAME: apnz-hz-cluster
          APINIZER_CACHE_CLUSTER_MEMBERS: 172.30.0.11,172.30.0.12,172.30.0.13
        env_file: [./secrets/cache.env]
        mem_limit: 4g
        restart: unless-stopped

      cache-2:
        image: apinizercloud/cache:2026.04.5
        container_name: apinizer-cache-2
        networks:
          apinizer:
            ipv4_address: 172.30.0.12
        environment: *cache-env
        env_file: [./secrets/cache.env]
        mem_limit: 4g
        restart: unless-stopped

      cache-3:
        image: apinizercloud/cache:2026.04.5
        container_name: apinizer-cache-3
        networks:
          apinizer:
            ipv4_address: 172.30.0.13
        environment: *cache-env
        env_file: [./secrets/cache.env]
        mem_limit: 4g
        restart: unless-stopped

    networks:
      apinizer:
        driver: bridge
        ipam:
          config:
            - subnet: 172.30.0.0/24
    ```

    #### Multi-Host

    On each host, set `APINIZER_CACHE_CLUSTER_MEMBERS` to the **same** comma-separated list of host IPs.

    <Tip>
    TCP/IP discovery is **symmetric** — if host B does not list host A, B will not accept A as a member even though A reaches B. Distribute the same peer list to every node.
    </Tip>

    ### 5. Verify

    REST health:

    ```bash
    curl -fsS http://127.0.0.1:8090/management/health
    ```

    Hazelcast members:

    ```bash
    docker logs apinizer-cache | grep -E 'Members \{'
    ```

    Expected on a 3-node cluster:

    ```
    Members {size:3, ver:3} [
        Member [172.30.0.11]:5701 - ...
        Member [172.30.0.12]:5701 - ...
        Member [172.30.0.13]:5701 - ...
    ]
    ```

    If only one member is listed, check firewall rules on `5701/tcp` and that every node lists the same peers.

    ### 6. Wire Workers to the Cache

    Each Worker must:
    - Use the **same** Hazelcast cluster name
    - Reach every Cache node on `5701/tcp`
    - See the same MongoDB database

    ### 7. Operations

    <CardGroup cols={2}>
      <Card title="Status">docker compose ps</Card>
      <Card title="Logs">docker compose logs -f cache</Card>
      <Card title="Restart">docker compose restart cache</Card>
      <Card title="Stop (drain)">docker compose stop -t 30 cache</Card>
    </CardGroup>

    ### 8. Resources

    | Workload | `mem_limit` |
    | --- | --- |
    | Single tenant, low traffic | 2 GB |
    | Production (1 Worker, ≤1 krps) | 4 GB |
    | Production cluster (3+ Workers) | 4–6 GB per node |

    Do not run the Cache with less than 2 GB in production.

  <Tab title="Integration Installation">
    ## Apinizer Integration

    The Apinizer **Integration** is a Quartz-based task-flow scheduler. Install it **only if** you use task flows or connector pipelines from the Manager UI.

    ### 1. Pull the Image

    ```bash
    VERSION=2026.04.5
    docker pull apinizercloud/integration:${VERSION}
    ```

    ### 2. Configure

    Required:

    | Variable | Description |
    | --- | --- |
    | `SPRING_DATA_MONGODB_URI` | MongoDB connection string — same instance the Manager uses |
    | `SPRING_DATA_MONGODB_DATABASE` | MongoDB database name — must match the Manager database |
    | `INTEGRATION_TIMEZONE` | IANA zone ID (e.g., `Europe/Istanbul`). Must match Manager and Worker timezone. |

    Optional:

    | Variable | Default | Description |
    | --- | --- | --- |
    | `SERVER_PORT` | `8092` | Spring Boot HTTP port |
    | `SPRING_PROFILES_ACTIVE` | `prod` | Keep on production hosts |
    | `JAVA_OPTS` | empty | Extra JVM flags |
    | `LOG_LEVEL` | `INFO` | Application log level |

    ### 3. Run the Container

    ```bash
    docker run -d \
      --name apinizer-integration \
      -p 8092:8092 \
      --memory=2g \
      --stop-timeout=120 \
      -e SPRING_DATA_MONGODB_URI='mongodb://user:pass@mongo:25080/?authSource=admin' \
      -e SPRING_DATA_MONGODB_DATABASE='apinizer' \
      -e INTEGRATION_TIMEZONE='Europe/Istanbul' \
      -e SPRING_PROFILES_ACTIVE=prod \
      -v apinizer-integration-logs:/app/logs \
      --restart unless-stopped \
      apinizercloud/integration:${VERSION}
    ```

    <Warning>
    Use `--stop-timeout=120` (or `stop_grace_period: 120s` in compose). Quartz is configured with `waitForJobsToCompleteOnShutdown=true`, so SIGTERM triggers a clean drain. Without the extended grace, Docker kills with SIGKILL after 10 seconds and in-flight task flows are cut mid-step.
    </Warning>

    Expected boot log:

    ```
    Integration timezone set to: Europe/Istanbul
    Initializing Custom MongoDB Job Store
    Quartz Scheduler Prod Environment Setup started
    MongoDB connection established for Quartz Job Store
    Tomcat started on port 8092 (http)
    ```

    ### 4. docker-compose

    ```yaml
    services:
      integration:
        image: apinizercloud/integration:2026.04.5
        container_name: apinizer-integration
        restart: unless-stopped
        mem_limit: 2g
        stop_grace_period: 120s
        ports:
          - "8092:8092"
        environment:
          SPRING_DATA_MONGODB_DATABASE: apinizer
          INTEGRATION_TIMEZONE: Europe/Istanbul
          SPRING_PROFILES_ACTIVE: prod
        env_file:
          - ./secrets/integration.env
        volumes:
          - integration-logs:/app/logs
        healthcheck:
          test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:8092/management/health | grep -q '\"status\":\"UP\"'"]
          interval: 30s
          timeout: 5s
          retries: 5
          start_period: 60s

    volumes:
      integration-logs:
    ```

    ### 5. Multi-Node / HA

    Multiple Integration containers can share the **same MongoDB database** — they form a clustered Quartz scheduler automatically. Each instance:
    - Auto-generates its `org.quartz.scheduler.instanceId`
    - Locks triggers via `schedule_integration_locks` so the same job never fires on two nodes at once
    - Recovers misfired jobs on startup

    <Tip>
    Run at least two Integration containers when you have time-critical scheduled flows.
    </Tip>

    ### 6. Wire the Manager to the Integration

    In the Manager UI (**Admin → Integration Servers**), add each Integration container's hostname/IP and HTTP port (`8092`). The Manager pushes job register/unregister and cache-refresh events via REST.

    ### 7. Verify

    ```bash
    curl -fsS http://127.0.0.1:8092/management/health
    ```

    Check the cluster picture from MongoDB:

    ```javascript
    use apinizer
    db.schedule_integration_scheduler_state.find().pretty()
    ```

    Each Integration node should have an entry with a recent `last_checkin_time`.

    ### 8. Operations

    <CardGroup cols={2}>
      <Card title="Status">docker compose ps</Card>
      <Card title="Logs">docker compose logs -f integration</Card>
      <Card title="Restart (drain)">docker compose restart -t 120 integration</Card>
      <Card title="Stop (drain)">docker compose stop -t 120 integration</Card>
    </CardGroup>

    ### 9. Resources

    | Workload | `mem_limit` |
    | --- | --- |
    | Few flows, light cadence | 1 GB |
    | Production (typical install) | 2 GB |
    | Heavy / many concurrent flows | 4 GB |

    <Note>
    For Integration troubleshooting, see [Troubleshooting](/en/setup/troubleshooting).
    </Note>

  <Tab title="API Portal Setup">
    ## Apinizer API Portal

    The Apinizer API Portal is the developer-facing UI — it lists APIs, manages subscriptions, and renders documentation. It connects to the Manager REST API for data; there is **no direct MongoDB connection** from the Portal.

    <Info>
    The Portal is stateless. Every container restart is safe and you can run multiple replicas behind a load balancer for horizontal scale.
    </Info>

    ### 1. Pull the Image

    ```bash
    VERSION=2026.04.5
    docker pull apinizercloud/apiportal:${VERSION}
    ```

    ### 2. Register the Portal in the Manager

    Before the container boots, register the Portal once in the Manager UI:

    1. **Portal → Portals → Create** in the Manager UI. Note the generated **Portal ID**.
    2. **Portal → API Keys → Generate** and copy the API key value (it is shown only once).

    ### 3. Configure

    Required:

    | Variable | Description |
    | --- | --- |
    | `API_PORTAL_ID` | Portal identifier from step 2 |
    | `API_PORTAL_MANAGEMENT_API_BASE_URL` | Manager REST API base URL (e.g., `https://manager.example.com`) |
    | `API_PORTAL_MANAGEMENT_API_KEY` | Manager API key from step 2 |

    Optional:

    | Variable | Default | Description |
    | --- | --- | --- |
    | `SERVER_PORT` | `8080` | HTTP port inside the container |
    | `SPRING_PROFILES_ACTIVE` | `prod` | Keep on production hosts |
    | `SSL_ENABLED` | `false` | Set `true` to terminate TLS inside the container |
    | `JAVA_OPTS` | empty | Extra JVM flags |

    ### 4. Run the Container

    ```bash
    docker run -d \
      --name apinizer-apiportal \
      -p 8081:8080 \
      --memory=2g \
      -e API_PORTAL_ID='portal-prod-01' \
      -e API_PORTAL_MANAGEMENT_API_BASE_URL='https://manager.example.com' \
      -e API_PORTAL_MANAGEMENT_API_KEY='<api-key>' \
      -e SPRING_PROFILES_ACTIVE=prod \
      -v apinizer-apiportal-logs:/app/logs \
      --restart unless-stopped \
      apinizercloud/apiportal:${VERSION}
    ```

    <Tip>
    Map the container's `8080` to host `8081` if the API Manager already binds host `8080`.
    </Tip>

    ### 5. docker-compose

    ```yaml
    services:
      apiportal:
        image: apinizercloud/apiportal:2026.04.5
        container_name: apinizer-apiportal
        restart: unless-stopped
        mem_limit: 2g
        ports:
          - "8081:8080"
        environment:
          API_PORTAL_ID: portal-prod-01
          API_PORTAL_MANAGEMENT_API_BASE_URL: https://manager.example.com
          SPRING_PROFILES_ACTIVE: prod
        env_file:
          - ./secrets/apiportal.env       # contains API_PORTAL_MANAGEMENT_API_KEY=…
        volumes:
          - apiportal-logs:/app/logs
        healthcheck:
          test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:8080/management/health | grep -q '\"status\":\"UP\"'"]
          interval: 30s
          timeout: 5s
          retries: 5
          start_period: 60s

    volumes:
      apiportal-logs:
    ```

    ### 6. Verify

    ```bash
    curl -fsS http://127.0.0.1:8081/management/health
    ```

    Open `http://<host>:8081/` in a browser. The first page load fetches the catalog from the Manager — if the Manager URL or API key are wrong, you will see a 502/timeout banner.

    ### 7. Operations

    <CardGroup cols={2}>
      <Card title="Status">docker compose ps</Card>
      <Card title="Logs">docker compose logs -f apiportal</Card>
      <Card title="Restart">docker compose restart apiportal</Card>
      <Card title="Stop">docker compose stop apiportal</Card>
    </CardGroup>

    ### 8. TLS Termination

    Two common patterns:

    1. **Reverse proxy in front (recommended)**: Terminate TLS at nginx, Traefik, or a load balancer; keep `SSL_ENABLED=false`.
    2. **In-container TLS**: Mount a PKCS#12 keystore and set:

       ```yaml
       environment:
         SSL_ENABLED: "true"
         SSL_KEY_STORE: /run/keystore/portal.p12
         SSL_KEY_STORE_TYPE: PKCS12
         SSL_KEY_STORE_PASSWORD: <password>
         SSL_KEY_ALIAS: apinizer
         SSL_PROTOCOLS: TLSv1.2,TLSv1.3
       volumes:
         - ./secrets/portal.p12:/run/keystore/portal.p12:ro
       ```

    ### 9. Horizontal Scale

    The Portal is stateless. To run multiple replicas:
    - Give each replica the **same** `API_PORTAL_ID`
    - Front them with a load balancer
    - Sticky sessions are **not** required

    For API Portal troubleshooting, see [Troubleshooting](/en/setup/troubleshooting).
    </Note>

  <Tab title="API Manager with SSL">
    ## Starting API Manager with HTTPS

    Apinizer API Manager can terminate HTTPS traffic inside the container. This is useful when running behind a CDN or in environments where reverse-proxy TLS termination is not available.

    ### Generate or Obtain a Certificate

    Create a PKCS#12 keystore (.p12) containing your certificate and private key:

    ```bash
    # Example: self-signed certificate (for testing only)
    keytool -genkey -alias apinizer -keyalg RSA -keystore manager.p12 \
      -keypass changeme -storepass changeme -storetype PKCS12 \
      -dname "CN=manager.example.com,O=Example,C=US" \
      -validity 365

    # Or convert an existing PEM certificate + key:
    openssl pkcs12 -export -in cert.pem -inkey key.pem \
      -out manager.p12 -name apinizer
    ```

    ### docker-compose with SSL

    ```yaml
    services:
      apimanager:
        image: apinizercloud/apimanager:2026.04.5
        container_name: apinizer-apimanager
        restart: unless-stopped
        mem_limit: 4g
        ports:
          - "8080:8080"
          - "8443:8443"        # HTTPS port
        environment:
          SPRING_PROFILES_ACTIVE: prod
          SPRING_DATA_MONGODB_DATABASE: apinizer
          SSL_ENABLED: "true"
          SSL_KEY_STORE: /run/keystore/manager.p12
          SSL_KEY_STORE_TYPE: PKCS12
          SSL_KEY_STORE_PASSWORD: changeme
          SSL_KEY_ALIAS: apinizer
          SSL_PROTOCOLS: TLSv1.2,TLSv1.3
          SERVER_PORT: "8443"
        env_file:
          - ./secrets/apimanager.env
        volumes:
          - ./secrets/manager.p12:/run/keystore/manager.p12:ro
          - apimanager-logs:/app/logs
        healthcheck:
          test: ["CMD-SHELL", "wget --no-check-certificate -qO- https://127.0.0.1:8443/management/health | grep -q '\"status\":\"UP\"'"]
          interval: 30s
          timeout: 5s
          retries: 5
          start_period: 90s

    volumes:
      apimanager-logs:
    ```

    Or use `docker run`:

    ```bash
    docker run -d \
      --name apinizer-apimanager \
      -p 8080:8080 \
      -p 8443:8443 \
      --memory=4g \
      -e SPRING_PROFILES_ACTIVE=prod \
      -e SPRING_DATA_MONGODB_DATABASE='apinizer' \
      -e SSL_ENABLED='true' \
      -e SSL_KEY_STORE='/run/keystore/manager.p12' \
      -e SSL_KEY_STORE_TYPE='PKCS12' \
      -e SSL_KEY_STORE_PASSWORD='changeme' \
      -e SSL_KEY_ALIAS='apinizer' \
      -e SERVER_PORT='8443' \
      -v ./secrets/manager.p12:/run/keystore/manager.p12:ro \
      -v apimanager-logs:/app/logs \
      --restart unless-stopped \
      apinizercloud/apimanager:${VERSION}
    ```

    ### Configuration Variables

    | Variable | Description |
    | --- | --- |
    | `SSL_ENABLED` | Set to `true` to enable HTTPS |
    | `SSL_KEY_STORE` | Path to PKCS#12 keystore inside container (recommended: `/run/keystore/<name>.p12`) |
    | `SSL_KEY_STORE_TYPE` | `PKCS12` (or `JKS` for legacy keystores) |
    | `SSL_KEY_STORE_PASSWORD` | Password to unlock the keystore |
    | `SSL_KEY_ALIAS` | Alias of the certificate entry in the keystore (default: `apinizer`) |
    | `SSL_PROTOCOLS` | Enabled TLS versions (e.g., `TLSv1.2,TLSv1.3`). Do not use TLSv1.0 or TLSv1.1 in production. |
    | `SERVER_PORT` | `8443` (HTTPS) or `8080` (HTTP default) |

    <Warning>
    Self-signed certificates are **only** suitable for testing. In production, use a CA-issued certificate and keep it up to date.
    </Warning>
