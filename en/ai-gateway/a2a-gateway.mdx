## Overview

**A2A (Agent2Agent)** is an open protocol that lets AI agents from different providers exchange tasks over a standard interface. Apinizer AI Gateway supports A2A **bidirectionally**:

- **Inbound Server** — Expose Apinizer as an A2A agent; external clients submit tasks via the `/a2a` endpoint
- **Outbound Client** — Send tasks to external A2A agents from Apinizer (the **A2A Agent Call** policy)

:::info

The A2A protocol is built on JSON-RPC 2.0. Apinizer supports both synchronous task submission and streaming responses via Server-Sent Events (SSE). For an introduction to agent-to-agent concepts, see [AI Fundamentals](/en/concepts/core-concepts/ai-fundamentals).

:::

## Inbound A2A Server

Publish Apinizer as an A2A agent so external clients can connect with any standard A2A client.

### Authentication

Each inbound agent uses one of two auth modes: **None** (an explicit no-auth opt-out) or **API Key** (a Bearer token compared against a configured secret). OAuth2 is reserved for a future release and can't be selected yet — an agent never advertises a security scheme the gateway can't actually enforce.

:::note
The API Key can reference an [environment variable](/en/concepts/core-concepts/environment-variable) (`${VARIABLE_NAME}`). If the variable can't be resolved when a request arrives, the key is treated as misconfigured and the request is denied — an unresolved placeholder is never compared as if it were the real key.
:::

### AgentCard

Every inbound agent publishes a self-describing **AgentCard**:

- Agent name, description, and capabilities
- Supported input/output modalities
- Authentication scheme — derived from the agent's own auth mode described above: no scheme when it's set to None, `bearerAuth` when it's set to API Key
- The AgentCard is published **unsigned** in this release

### Task Submission

Clients submit tasks with the `message/send` method:

1. The client starts a task (which may contain text, files, or structured data)
2. Apinizer **accepts** the task and returns a task ID
3. The client polls for task status
4. Once completed, the result is returned in the task object

### Task States

| State | Description |
|-------|-------------|
| `submitted` | Task received, processing not yet started |
| `working` | Task is being processed |
| `input-required` | Agent is waiting for additional input |
| `completed` | Task completed successfully |
| `failed` | Task ended with an error |
| `canceled` | Task was canceled |

:::note

Task records are cleaned up automatically some time after reaching a **terminal** state (completed/failed/canceled); active tasks are retained indefinitely.

:::

### Task Content

A task's parts aren't limited to plain text — the current release supports **image and structured data parts** alongside text, so an agent can send or receive an image or a JSON payload as part of a task, not just a text string.

## Streaming Responses

Clients can request streaming task updates over SSE, delivered chunk by chunk. If the connection drops, the client can **resubscribe** and continue from where it left off (replayed from the stored task history).

## Push Notifications

For long-running tasks, clients can configure **webhook-based push notifications** — when a task completes or fails, Apinizer sends a notification request (with an authentication token) to the configured webhook URL. This is the **primary completion channel** for long-running agent tasks, in preference to polling.

## Outbound A2A Client

To send tasks from Apinizer to external A2A agents:

<Steps>
  <Step title="Create an A2A Connection">
    Go to **Management** → **Connections** → **A2A Agent Connection** and define the target agent's endpoint address and authentication — including OAuth2 client-credentials for agents that require it.
  </Step>

  <Step title="Test the Connection">
    Before saving, select the environment you plan to deploy to and click **Test Connection**. The check is a live round-trip through that environment's Worker — not the Manager — so a success means the Worker itself can reach the target agent, which is what matters at request time.
  </Step>

  <Step title="Fetch the AgentCard">
    Once the connection is saved, Apinizer automatically fetches and previews the target agent's **AgentCard** — capabilities and supported modalities are validated.
  </Step>

  <Step title="Add the A2A Agent Call Policy">
    Add the A2A agent call policy to your API proxy and select the target A2A connection.
  </Step>

  <Step title="Monitor Tasks">
    Use the **A2A Task Monitor** screen to watch the status of submitted tasks (submitted/working/completed/failed) in real time.
  </Step>
</Steps>

### Outbound Authentication

Outbound A2A connections are configured to match whatever auth scheme the target agent requires; you can connect to targets that require a token, including an **OAuth2 client-credentials** flow — the connection acquires the token automatically at runtime.

## Security: Egress Protection

Outbound A2A calls (AgentCard fetch + task submission) are protected against **SSRF (Server-Side Request Forgery)**:

:::danger

The target URL is validated against **private IP ranges** (including cloud metadata endpoints) and **localhost** addresses. Requests to these addresses are blocked.

:::

This protection prevents a malicious or misconfigured A2A connection from reaching internal services on the network where Apinizer runs. The same protection is shared by [MCP Gateway](/en/ai-gateway/mcp-gateway) outbound connections.

## Limitations

:::note
Signed AgentCards and human-in-the-loop task approval are out of scope for this release. A federated agent registry is not planned.
:::
