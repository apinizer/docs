## Failover Chains

When you configure an AI proxy, you can list more than one provider connection for a model — a **primary** and one or more **backups**. If a request to the primary fails (timeout, error response, or the primary is unavailable), Apinizer automatically retries it against the next connection in the chain, without the client seeing the failure.

```mermaid
sequenceDiagram
    participant Client as 👤 Client
    participant Gateway as 🚪 AI Gateway
    participant Primary as 🅰️ Primary Provider
    participant Backup as 🅱️ Backup Provider

    Client->>Gateway: Chat completion request
    Gateway->>Primary: Forward request
    Primary--xGateway: Timeout / error
    Gateway->>Backup: Retry on next connection in chain
    Backup->>Gateway: Response
    Gateway->>Client: Response + usage metrics
```

## Cost-Aware and Latency-Aware Routing

Two of the six strategies route dynamically instead of following a fixed order:

- **Least-Cost** routing looks up each candidate connection's price in the [Model Catalog](/en/ai-gateway/model-catalog) and sends the request to the cheapest option.
- **Least-Latency** routing uses periodic health-check probes against each connection and sends the request to the fastest responder; a connection with no recent probe result is skipped for this comparison rather than blocking the request.

## Per-Leg Cost Cap

You can set a maximum cost for an individual failover leg. Before forwarding a request to a connection, Apinizer estimates its cost from the input tokens and the requested maximum output tokens, using catalog pricing. If the projected cost exceeds that leg's cap, the leg is skipped and the next one in the chain is tried. A connection whose model isn't in the pricing catalog is **not** skipped by this check — the request goes through rather than being blocked by an unknown price.

## Reliable Retries and Billing

Failover retries are billed correctly: a failed attempt against one connection is never counted against your usage or budget — only the connection that actually served the response is billed and logged.

## Tool-Call Loop (Agentic Requests)

When a model's response includes a tool call, Apinizer can dispatch it to the tools configured for the proxy and feed the result back to the model automatically, continuing the exchange until the model returns a final answer or a configurable turn limit is reached (5 turns by default). Usage and cost are tracked per turn, so multi-step tool use shows up in reports the same way a single request would.

## Reusable Failover Templates

Rather than re-entering the same primary/backup chain in every AI proxy, you can define it once as a **failover template** and reference it from any number of proxies in the project. A template carries exactly what an inline chain does — the ordered list of provider connections (each with its own optional model, credential, timeout, per-leg cost cap, and weight), the failover strategy, the retryable status codes, the retry count, and the Conditional-strategy latency threshold.

Manage templates from **AI Gateway → Failover Templates** in the project menu, or with APIops:

```
GET/POST/PUT/DELETE /apiops/projects/{projectName}/ai-failover-templates/{templateName}/
```

### Referencing a Template

An AI proxy's routing configuration has a **Failover Template** field, in the same dialog where you'd otherwise build an inline chain (**Provider Failover Chain**). Selecting a template clears the inline chain — the two are mutually exclusive in the editor, and the chain-entry editor is disabled while a template is referenced. Clear the template reference to go back to authoring an inline chain.

:::note Templates are expanded at deploy time, not resolved at request time
The gateway worker has no notion of failover templates. When an AI proxy that references a template is deployed, Apinizer copies the template's chain, strategy, status codes, retry count, and latency threshold into that deployment's snapshot — the worker only ever sees a concrete chain, identical to one you'd have entered inline.

Because the copy happens at deploy time, editing a template does not retroactively change the behavior of proxies that already reference it. **Redeploy each referencing proxy** to pick up the template's latest configuration.
:::

### Deactivating and Deleting a Template

A template can be deactivated without deleting it. Deploying a proxy that references an inactive (or missing) template fails immediately with an error — Apinizer never silently deploys a proxy with an effectively empty failover chain.

A template that's still referenced by one or more AI proxies **cannot be deleted**. The delete confirmation shows how many proxies currently reference it; repoint or clear those references first.
