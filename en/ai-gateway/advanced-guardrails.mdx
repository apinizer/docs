## Overview

In addition to the core PII masking and prompt guard, Apinizer AI Gateway offers four new guardrail layers:

- **DLP Guard** — Prevents secrets (API keys, private keys, etc.) and other sensitive data from leaking in requests/responses
- **Oversized Guard** — Rejects excessively large requests (by token count or character count)
- **Loop Guard** — Detects the same request being repeated within a short time window
- **Off-Topic Guard** — Flags/blocks requests outside the gateway's intended topic scope

## DLP (Data Loss Prevention) Guard

The DLP guard scans the request and/or response body against **known sensitive-data patterns**.

### Supported Built-In Patterns

| Pattern | Example |
|---------|---------|
| AWS access key | `AKIAIOSFODNN7EXAMPLE` |
| OpenAI API key | `sk-...` |
| GitHub personal access token | `ghp_...` |
| PEM private key block | `-----BEGIN PRIVATE KEY-----` |
| JWT (JSON Web Token) | `eyJhbGci...` |

:::info

Patterns ship as ready-made **presets**; you can also define additional custom patterns (regex) for your organization.

:::

### Direction and Action

The DLP guard can run in either **request** or **response** direction, or **both**, and applies one of three actions on a match:

- **BLOCK** — The request/response is rejected entirely
- **FLAG** — Processing continues, but the event is flagged in logs/reports
- **MASK** — The matched value is replaced with `[REDACTED]`, processing continues

:::warning

**In streaming responses**: the DLP guard scans the response in windows (chunk groups). Once a **BLOCK** match is found in one window, all **remaining windows** of the stream are automatically masked as well (a sticky/latched block) — this prevents a secret split across chunk boundaries from leaking.

:::

## Oversized Guard

Limits request size along two dimensions:

- **Max tokens per request** — The request is rejected if the estimated prompt token count exceeds this value
- **Max prompt characters** — The request is rejected if the raw prompt text exceeds the character limit

This guard prevents uncontrolled growth in cost and resource consumption from oversized requests; a general request-body size cap is also enforced at the gateway level.

## Loop Guard

Detects the **same request** being sent repeatedly within a short time window (e.g. a client-side retry loop gone wrong):

- A cryptographic hash of the request content is computed
- The number of repeats of the same hash is tracked over a configurable time window (N seconds, M requests)
- If the threshold is exceeded, the request is **flagged (FLAG)** or **blocked (BLOCK)**

:::tip

The loop guard is especially useful for catching unexpected cost spikes early in autonomous agent chains, where an agent may repeatedly regenerate the same request.

:::

## Off-Topic Guard

Detects requests outside the gateway's intended usage scope — for example, preventing a customer-support bot from being used for general chit-chat or unrelated topics:

<Steps>
  <Step title="Define Topics">
    An administrator defines the topics within the gateway's scope (e.g. "billing inquiries", "product support requests").
  </Step>

  <Step title="Compute Embeddings">
    Each topic definition is converted into an embedding vector.
  </Step>

  <Step title="Compare Similarity">
    Every incoming request is converted to an embedding and compared against the defined topics using **cosine similarity**.
  </Step>

  <Step title="Threshold Decision">
    If the similarity is below the threshold, the request is considered off-topic and the configured action is applied.
  </Step>
</Steps>

:::note

The off-topic guard runs in **ASYNC** mode by default — it is evaluated in the background without slowing down the request path, and reported; a synchronous (guaranteed-blocking) mode is also configurable.

:::

## Configuration

<Steps>
  <Step title="Add the Policy">
    Add the desired guard policy (DLP / Oversized / Loop / Off-Topic) to the relevant API proxy or policy group.
  </Step>

  <Step title="Select Patterns / Thresholds">
    For DLP, select the built-in pattern sets (AWS/OpenAI/GitHub/PEM/JWT) or define your own pattern; for the other guards, enter the threshold and time-window values.
  </Step>

  <Step title="Choose the Action">
    Choose which action to apply: BLOCK / FLAG / MASK.
  </Step>

  <Step title="Deploy">
    After saving the policy, click **Deploy** — saving and deploying are separate steps; there is no automatic deployment.
  </Step>
</Steps>
