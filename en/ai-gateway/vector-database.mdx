## Overview

A vector database stores the embeddings generated from your [knowledge bases](/en/ai-gateway/knowledge-bases) and is what [RAG](/en/ai-gateway/rag-injection) and the [semantic cache](/en/ai-gateway/semantic-cache) search against. Apinizer connects to it as an ordinary connection, the same way it connects to other backing services.

## Choosing a Backend

Three interchangeable backends are supported — pick whichever fits your deployment:

<CardGroup cols={1}>
  <Card title="PostgreSQL / pgvector" icon="database">
    A good fit if you already run PostgreSQL and want vector search alongside your existing data.
  </Card>
  <Card title="Qdrant" icon="boxes">
    A purpose-built vector database for teams that want a dedicated similarity-search engine.
  </Card>
  <Card title="Redis" icon="zap">
    A good fit if you already run Redis and want fast, in-memory similarity search.
  </Card>
</CardGroup>

Knowledge bases, RAG, and the semantic cache all work the same way regardless of which backend you choose — the backend is a deployment choice, not a feature trade-off.

## Embedding Provider

A single, configurable **embedding provider** (for example OpenAI, Voyage, or a self-hosted Ollama model) generates the vector representations used both when a knowledge base is indexed and when a request is matched against it. Using one provider consistently for both sides keeps indexed content and incoming requests comparable.

:::note
Before an embedding request is routed to a provider, Apinizer verifies that the provider actually supports embeddings — so a request never gets routed to a provider that can't generate them.
:::

## Tenant Isolation

Every similarity search is filtered to the requesting project's boundary, the same isolation guarantee described in [RAG scope isolation](/en/ai-gateway/rag-injection#scope-isolation) — one tenant's search can never return another tenant's content, regardless of which backend is in use.
