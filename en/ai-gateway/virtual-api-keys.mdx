## What Is a Virtual API Key

A virtual API key is the credential your applications actually use — it is **not** the real OpenAI, Anthropic, or other provider key. Apinizer maps the virtual key to one or more real provider connections behind the scenes, so a client only ever needs one key regardless of how many providers you connect.

Clients pass the virtual key exactly where they'd pass a normal OpenAI API key:

```python
from openai import OpenAI

client = OpenAI(
    api_key="your-virtual-api-key",
    base_url="https://your-apinizer-gateway.com/api/ai/v1"
)
```

## Scopes

A virtual key is issued against one of four scopes:

<CardGroup cols={2}>
  <Card title="User" icon="user">
    Tied to a single person's credential — usage and quota are attributed to that user.
  </Card>
  <Card title="Role" icon="shield">
    Shared by everyone holding a given role — useful for team-wide tooling with a per-role budget.
  </Card>
  <Card title="Project" icon="folder">
    Scoped to a project, so every application inside that project shares one key and one quota.
  </Card>
  <Card title="Team" icon="users">
    Shared across a team, independent of project boundaries.
  </Card>
</CardGroup>

## Why Use a Virtual Key

- **No real key leakage** — client applications, logs, and network traces never see the underlying provider credential.
- **Central control** — revoke or rotate access from one place without redeploying client applications.
- **Attribution** — usage, cost, and quota reports (see [Reports and Analytics](/en/ai-gateway/reports)) are broken down by the scope the virtual key belongs to.

## Creating a Virtual API Key

<Steps>
  <Step title="Open Virtual Keys">
    In the Apinizer UI, go to **Admin** → **AI Gateway** → **Virtual API Keys**.
  </Step>

  <Step title="Click Add Key">
    Choose the scope (User, Role, Project, or Team) and the target it applies to.
  </Step>

  <Step title="Restrict Access (optional)">
    Limit which provider connections or models this key is allowed to reach.
  </Step>

  <Step title="Save">
    Click **Save**. The key is issued immediately and ready for clients to use as `api_key`.
  </Step>
</Steps>

## Rotating a Key Without Downtime

Rotating a virtual key doesn't require a coordinated client deploy:

1. Issue a new key for the same scope.
2. The previous key stays valid through a **grace period** you set (an expiry date on the old key), so in-flight and slow-to-update clients keep working.
3. The new key takes priority for new requests immediately.
4. Once the grace period ends, the old key stops working automatically.

The list and update screens for virtual keys guide you through this rotation flow step by step.
