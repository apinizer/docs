## Overview

An **LLM Provider Connection** is a secure, encrypted link to an external LLM service (OpenAI, Anthropic, Azure, etc.). Each connection holds the provider type, API credentials, and deployment metadata.

## Provider Types

Apinizer AI Gateway supports:

| Provider | Type | Deployment (default) | Notes |
|----------|------|----------------------|-------|
| OpenAI | `OPENAI` | Cloud | GPT-4, GPT-4 Turbo, GPT-3.5, etc. |
| Anthropic | `ANTHROPIC` | Cloud | Claude models |
| Azure OpenAI | `AZURE` | Cloud | OpenAI models hosted on Azure |
| Google Vertex AI | `VERTEX` | Cloud | PaLM 2, Gemini |
| AWS Bedrock | `BEDROCK` | Cloud | Claude, Llama, Mistral |
| vLLM | `VLLM` | On-Premise | Self-hosted inference engine |
| Ollama | `OLLAMA` | On-Premise | Local LLM inference |
| Custom OpenAI-Compatible | `CUSTOM_OPENAI_COMPAT` | On-Premise | Any OpenAI-compatible endpoint |

## Deployment Type

Each connection declares whether the LLM runs in the cloud or on-premise:

- **Cloud** — LLM is managed by the provider (OpenAI, Anthropic, Azure, Vertex, Bedrock)
- **On-Premise** — LLM runs in your infrastructure (vLLM, Ollama, custom endpoint)

**Deployment type is auto-filled** based on provider type when you create a connection, but you can override it if needed (e.g., using a custom vLLM instance on a cloud provider).

## Creating a Connection

<Steps>
  <Step title="Navigate to Connections">
    In the Apinizer UI, go to **Admin** → **API Gateway** → **LLM Providers**.
  </Step>
  
  <Step title="Click Add Provider">
    Click the **Add** button to open the provider form.
  </Step>
  
  <Step title="Select Provider Type">
    Choose the provider from the dropdown (OpenAI, Anthropic, Azure, vLLM, Ollama, etc.).
    
    Deployment type **auto-fills**:
    - If provider is OpenAI, Anthropic, Azure, Vertex, or Bedrock → defaults to **Cloud**
    - If provider is vLLM, Ollama, or Custom → defaults to **On-Premise**
  </Step>
  
  <Step title="Enter Credentials">
    Paste the API key or secret. Apinizer **encrypts and never logs** credentials.
    
    Examples:
    - **OpenAI**: paste your `sk-...` key
    - **Anthropic**: paste your `sk-ant-...` key
    - **Azure**: paste the endpoint URL and key
    - **vLLM/Ollama**: paste the base URL (e.g., `http://localhost:8000`)
  </Step>
  
  <Step title="Override Deployment Type (optional)">
    If your cloud provider has an on-premise instance (or vice versa), toggle the **Deployment Type** dropdown to Cloud or On-Premise.
  </Step>
  
  <Step title="Save">
    Click **Save**. The connection is encrypted and stored. Credentials are never visible in logs or exports.
  </Step>
</Steps>

## Using Connections in Routing

Once created, a provider connection is available for **API Proxy routing**. When you create an API proxy, you assign models and select which connection to use:

```yaml
# Example: route requests for "gpt-4o" to OpenAI connection
Proxy: "my-chat-api"
Models:
  - name: "gpt-4o"
    provider: "OpenAI" # reference the saved connection
  - name: "gpt-4-turbo"
    provider: "OpenAI"
  - name: "claude-3-sonnet"
    provider: "Anthropic" # different connection
```

## Monitoring Connections

- **Usage by Deployment** — See how many tokens were consumed by cloud vs. on-premise connections in **AI Reports** → **By Deployment Type**
- **Cost Attribution** — Each connection logs its deployment type in traffic records for accurate cost allocation
- **Failover & Load Balancing** — Multiple connections to the same provider can be configured for redundancy (advanced)

## Troubleshooting

**Connection Test Failed**
- Verify the API key or endpoint URL is correct
- Check that credentials have the required permissions
- Ensure the provider service is accessible from Apinizer's network

**Credentials Not Saving**
- Ensure you have admin permissions
- Clear browser cache and try again

**Wrong Deployment Type Selected**
- Edit the connection and toggle **Deployment Type** to the correct value
- Deployment type changes are retroactive (logs will show the updated type going forward)
