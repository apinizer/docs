## AI Permission Groups

Access to AI Gateway is scoped through three permission groups, each independent of the others:

<CardGroup cols={1}>
  <Card title="AI Development" icon="wrench">
    Create and edit AI proxies, policies, provider connections, virtual API keys, and the model catalog.
  </Card>
  <Card title="AI Budgets" icon="dollar-sign">
    Manage token quotas, monthly USD budgets, and per-model pricing.
  </Card>
  <Card title="AI Analytics" icon="chart-bar">
    View usage reports, cost breakdowns, and request traces — read-only.
  </Card>
</CardGroup>

## AI Roles

Four built-in roles combine these permission groups so you can assign access by job function instead of granting each permission individually:

| Role | Development | Budgets | Analytics |
|------|:---:|:---:|:---:|
| **AI Admin** | ✅ | ✅ | ✅ |
| **AI Developer** | ✅ | – | – |
| **AI Finance** | – | ✅ | – |
| **AI Analytics** | – | – | ✅ |

Assign these roles to users the same way you assign any other Apinizer role — a person can hold a different AI role in each project or team they belong to.

## Explicit Deploy (Save Isn't Deploy)

Saving a change to an AI proxy, policy, or provider connection doesn't put it into effect immediately — the change sits in a **redeploy required** state until someone explicitly deploys it. This gives you a review point between editing and going live, the same explicit-deploy model used throughout the Apinizer gateway. See [Deployment Models](/en/concepts/deployment/deployment-models) for how deployment works across environments.

## Duplicate and Rollback via Export/Import

AI global policies can be exported and re-imported, the same portability mechanism used elsewhere in Apinizer:

- **Duplicate** a working configuration into another project or environment by exporting it and importing it there
- **Roll back** to a previous state by re-importing an earlier export

See [Global Policies](/en/develop/global-configurations/global-policies) for the shared policy model this applies to.

## Directory (LDAP/AD) Identity Sync for AI Users

If your organization manages identities in LDAP or Active Directory, those users — and the AI roles assigned to them — can be kept in sync automatically instead of maintained by hand in Apinizer. This uses the same directory sync integration as the rest of Apinizer's identity management, so AI users don't need a separate onboarding process. See [LDAP / Active Directory](/en/admin/identity-management/identity-providers/ldap-active-directory) to configure directory sync.
