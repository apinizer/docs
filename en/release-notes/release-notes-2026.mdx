## Version 2026.07.0

:::warning

**IMPORTANT — Action required if you use the SQL/JDBC traffic-log connector:** If your Apinizer installation writes API proxy traffic logs to a relational database (Oracle, MySQL, PostgreSQL, or SQL Server) via the SQL/JDBC traffic-log connector, you must run a log-table ALTER script **before or during** the upgrade to 2026.07.0. This adds the new AI Gateway, MCP Gateway, A2A Gateway, and routing-diagnostics columns to `log_ApiProxyTraffic`. **This step is not required if you use the Elasticsearch or MongoDB traffic-log connector** — those already carry the new columns automatically. [Log Table ALTER Scripts](/en/release-notes/2026-07-0-log-table-alter)

:::

:::note

The main theme of this release is the Apinizer **AI Gateway** module — bringing LLM-backed services under the same lifecycle, routing, security, and observability model as regular APIs.

:::

<span style={{ display: 'inline-block', backgroundColor: '#86efac', color: '#056d4f', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>FEATURED NEW FEATURES</span>

- ### AI Gateway — Manage LLM Services Like Any Other API
A full AI Gateway module has been added alongside the existing API proxy model: create an "AI" type API proxy that fronts LLM services behind a single OpenAI-compatible endpoint, backed by provider adapters for OpenAI, Azure OpenAI, Gemini, Bedrock, Vertex, DeepSeek, Moonshot, and Ollama (plus any other OpenAI-compatible self-hosted engine), with full request/response streaming (SSE) support end-to-end. [See.](/en/ai-gateway/overview) APNZ-5128

- ### Multi-Provider Routing and Automatic Failover
AI proxies can route across providers by cost or latency, balance traffic across an equal-priority pool of N primary connections (round-robin, weighted, random, or LRU), and fail over automatically through a configured backup chain when a primary connection errors out or becomes unavailable. A reusable Failover Template lets a chain be defined once and referenced by name from any number of AI proxies — the template is expanded into the proxy's own routing configuration at deploy time. [See.](/en/ai-gateway/routing-and-failover) APNZ-5128

- ### AI Guardrail Policies — Prompt, PII, DLP, Topic, and Loop Protection
A family of AI-specific guardrail policies protects request and response traffic: Prompt Guard detects jailbreak and prompt-injection attempts (evaluated inline, asynchronously, or in shadow mode), PII Mask redacts personal data such as national ID numbers, IBANs, and phone numbers in both regular and streamed responses, DLP Guard blocks, flags, or masks matches against configurable secret/sensitive-data patterns, Topic Guard keeps conversations on an approved subject using embedding similarity, Context Integrity guards against conversation tampering, and Loop Guard curbs runaway agentic tool-call loops. Guardrail presets can be defined once per project and reused across AI proxies. [See.](/en/ai-gateway/advanced-guardrails) APNZ-5128

- ### Semantic Cache for LLM Responses
AI proxies can cache LLM responses at two layers — an exact-match cache for identical prompts and a vector-similarity semantic cache for near-duplicate prompts — cutting repeated cost and latency for common questions before a request ever reaches the provider. [See.](/en/ai-gateway/semantic-cache) APNZ-5128

- ### Token and Cost Budgets with Rate Limiting
Token and USD budgets can be enforced at project, application, credential, and organization scope, with pre-flight reservation against the estimated request cost and a reconciliation pass against the real usage once a response (streamed or unary) completes. Soft caps warn while still letting traffic through; hard caps reject further requests once a quota is exhausted. [See.](/en/ai-gateway/token-quotas) APNZ-5128

- ### Retrieval-Augmented Generation (RAG) and Knowledge Bases
Knowledge bases can be built from your own documents (PDF ingestion with chunking and de-duplication) and indexed into a vendor-agnostic vector store — pgvector, Qdrant, or Redis — using an embedding provider such as Ollama or OpenAI. A RAG injection policy retrieves the relevant passages at request time and grounds the model's response in them, and a knowledge base can be reindexed on demand as its documents change. [See.](/en/ai-gateway/rag-injection) APNZ-5128

- ### MCP Gateway — Inbound and Outbound Model Context Protocol
Apinizer now participates on both sides of MCP (Model Context Protocol): an inbound MCP server exposes any AI proxy as a discoverable, callable tool set (`initialize`/`tools/list`/`tools/call`) for MCP-compatible agents, while an outbound MCP Tool Call policy lets an AI flow call tools on an external MCP server as part of handling a request. [See.](/en/ai-gateway/mcp-gateway) APNZ-5128

- ### A2A Gateway — Inbound and Outbound Agent2Agent Protocol
Apinizer AI Gateway also supports A2A (Agent2Agent) bidirectionally: inbound, an AI proxy can be published as an A2A agent with an AgentCard and v1.0 SendMessage/GetTask task lifecycle, and outbound, an A2A Agent Call policy lets an AI flow delegate work to an external A2A agent as an agentic tool call. [See.](/en/ai-gateway/a2a-gateway) APNZ-5128

- ### LLM Providers, Model Catalog, and Credential-Based Access
LLM provider connections store their API keys as encrypted secrets and can be reused across any number of AI proxies, a model catalog tracks each model's pricing and capabilities for cost tracking and routing decisions, and consumer-facing access is scoped through credentials — so a given application or user can be granted its own model and budget access without touching anyone else's. [See.](/en/ai-gateway/llm-providers) APNZ-5128

- ### AI Analytics and Observability
An Elasticsearch-backed AI analytics layer tracks every AI request with dedicated fields for provider, model, token counts, cost, latency, cache hits, and guardrail hits — including for streamed responses, whose tokens are accounted for once the stream completes. Dashboards surface KPIs and token/cost trends, provider and model share, guardrail-hit and cache-efficiency reports, and usage breakdowns by person, group, organizational unit, application, and model, alongside a live request trace view. [See.](/en/ai-gateway/reports) APNZ-5128

- ### AI Assets Manageable Through APIops
Every AI asset type — LLM provider connections, AI proxies, MCP and A2A connections, knowledge bases, vector database connections, budgets, and failover templates — can now be created, updated, and deleted by name through APIOps, the same as regular API proxies and connections. APNZ-5128

- ### Portal Applications in Identity Management with Metadata and AI Budget Scoping
Portal Applications can now be viewed and managed by admins directly from the Identity Management menu across all projects, with a dedicated read-only view page and Edit/Delete actions, alongside a new encrypted key-value Metadata tab shared with the self-service Portal UI. AI token/cost budgets can now be scoped down to an individual application's credential without breaking existing organization-wide budgets left unset. A follow-up fix also closed a data-sync bug where legacy Portal Applications with a null portal association were invisible from the Portal UI while still showing in Manager. [See.](/en/admin/identity-management/portal-applications) APNZ-6419

- ### API Portal Support Tickets
End users can now open, message on, and track support tickets directly from the API Portal. API Manager gains a matching admin-side ticket inbox — list, detail view, KPI dashboard, custom ticket-number prefixing, and an unassigned-ticket count — with an optional Jira status mapping that can drive the ticket lifecycle (Open, Waiting for Customer, Answered, Resolved). APNZ-6468

<span style={{ display: 'inline-block', backgroundColor: '#bfdbfe', color: '#1e40af', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>IMPROVEMENTS</span>

- ### Backend Network Sub-Phase Timing Metrics and Routing Diagnostics Panel
The gateway now measures DNS lookup, TCP connect, TLS handshake, time-to-first-byte, and body-read duration for every backend HTTP call, recording them alongside existing routing trace data. These sub-phase timings are surfaced in a Routing Diagnostics panel, giving support and operations teams a way to pinpoint where a slow or timed-out backend call actually stalled without an external APM trace. [See.](/en/operations/common-problems-solutions/routing-issues) APNZ-6345

- ### Backend Routing Address HTTP Method Override
Each backend routing address definition can now optionally pin its own HTTP method instead of always forwarding the method the request arrived with. The override is resolved per address — primary, canary, sticky, load-balanced, and failover each apply their own configured value — while mirrored traffic keeps using its own address override. The gateway also now recognizes the HTTP QUERY method end-to-end: a client request using QUERY is accepted and forwarded to the backend with its body intact, instead of failing with a 500 error as before. [See.](/en/develop/routing/http-routing) APNZ-6430, APNZ-6401

- ### Attribution-Enriched Quota Usage Tracking and Alerting
A new quota usage snapshot subsystem periodically resolves raw gateway quota counters (credential plan, API-based quota policy, Rate Limit Control List) back to their credential, app, product/account, and organization. This snapshot now powers both the Quota Usage & Alerts consumption report and configurable quota-threshold email alert definitions. [See.](/en/admin/identity-management/quota-usage-and-alerts) APNZ-6431

- ### Key Store, Trust Store, and Key Export/Import
Key Store and Trust Store definitions (JKS/PFX), plus private/public/secret key definitions, can now be exported and imported the same way certificates already could — as a binary file or a portable JSON export. The same operations were added to APIops and wired into the Manager UI on both the Key Store and Crypto Key screens. [See.](/en/admin/secrets-management/keystore-management) APNZ-4959, APNZ-6337, APNZ-6338, APNZ-6339

- ### Configurable Scope-Mismatch Behavior for OAuth2/JWT Token Endpoints
Token issuance for credential-based and proxy-based OAuth2/JWT flows now resolves scope mismatches through a shared, centrally configurable setting instead of always failing with a 401 invalid_scope error. Administrators can choose to issue a token with the requested∩granted scope intersection instead of rejecting the request outright, closing a prior gap where the proxy-side client_credentials flow performed no scope validation at all. [See.](/en/admin/system-settings/token-management-settings) APNZ-6232

- ### Keystore Studio
Apinizer Toolbox gained a new "Keystore Studio" tool — a browser-based counterpart to desktop KeyStore Explorer. It opens and inspects JKS/PKCS12 keystores, generates RSA/EC key pairs with self-signed certificates, imports PKCS#12/PKCS#8/OpenSSL/pasted-PEM/trusted certificates, and can save the result directly into the Apinizer secret pool. [See.](/en/admin/secrets-management/keystore-management) APNZ-6308

- ### Multi-Value Project Tags with Tag-Based Filtering
Projects can now be tagged with multiple free-text labels from the project definition screen, and the project selection popup lets users filter the project list by these tags — making it easier to organize and locate projects in organizations with many of them. [See.](/en/admin/project-management/project-creation) APNZ-4954

- ### Optional WS-Addressing Fields in SOAP Method Settings
The WSA Settings section in SOAP Method Settings no longer requires every WS-Addressing address field to be filled in before saving; these 8 fields are now optional, while the mandatory enum selects and the top-level soapAction remain required. [See.](/en/develop/api-proxy-configuration/soap-method-configuration) APNZ-6247

- ### Task Flow Duplicate Action
A Duplicate action has been added to the row-level menu on the Task Flow Manager list, letting users clone an existing task flow in one click. The duplicated flow is issued a brand-new execution token rather than inheriting the source flow's token, so the copy cannot be invoked with the original flow's credentials. [See.](/en/integrations/api-integrator-task-flow-builder/task-flow-design) APNZ-4840

- ### Duplicate-Email Prevention and Password Validation for Organization Manager Portal Accounts
The Organization Manager Portal account create/edit screen now blocks Save whenever the entered email address already belongs to another account, checked live as the field is edited. The form also enforces password complexity — minimum/maximum length and a mix of uppercase and lowercase characters. APNZ-5941

<span style={{ display: 'inline-block', backgroundColor: '#e65a2a', color: '#ffffff', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>FIXES</span>

- ### Fixed False-Positive Application Log Errors for Successful DB-2-API Stored Procedure Calls
A successful DB-2-API stored procedure call invoked through Try It could still be recorded as an Error in the Application Logs, because the log result was derived from the procedure's inner success flag instead of the actual outcome. Try It and API Proxy invocations of the same procedure now log consistently. [See.](/en/tutorials/api-creator-scenarios/db2api-stored-procedure-calling) APNZ-6086

- ### Fixed FAQ Entries Being Deleted When Edited in the Portal
Editing an existing Portal FAQ entry and saving could wipe the record instead of updating it, because the FAQ DTO did not carry the portal association across the entity/DTO round-trip. The DTO now maps the portal association in both directions. APNZ-6095

- ### Fixed Empty Accordion Boxes on Published Portal Pages
HTML content pages using accordion-style boxes rendered correctly in the Manager's preview but showed empty boxes once published, because the lazy-render script was duplicated across components and not reliably re-triggered. Script execution has been consolidated into a single directive invoked from each relevant Portal component. APNZ-6182

- ### Fixed Portal API Specification Examples Rendering as Raw Java Output Instead of JSON
Sample request and response bodies on Portal API specification pages appeared as raw Java Map output instead of valid JSON. Example values are now properly JSON-serialized for both request and response examples, matching what the Manager UI already showed. APNZ-6356, APNZ-6364

- ### Fixed Portal Reporting and Dashboard Issues Found in QA
Several Portal Reports/Analytics issues were fixed: the API test console can now be disabled with an explanatory message when a consumer has no subscription, the analytics dashboard no longer shows stale data after switching accounts, multi-select filters show defensive labels instead of raw object text, and the Portal management API's analytics endpoints now enforce portal-account scoping, closing a cross-portal data leak. APNZ-6432

- ### Fixed Alert/Log Retention Purge Job Running at the Wrong Time After Scheduler Reload
The Quartz cron trigger for retention/purge jobs lost its configured timezone whenever the scheduler reloaded triggers from the database, causing jobs to fire in the JVM's default timezone instead of the configured one. The trigger's timezone is now persisted and restored on reload, with a backward-compatible fallback for older trigger records. [See.](/en/operations/database-growth-management/alert-history-collection-cleanup) APNZ-5931

- ### Fixed Inaccurate WSDL Reparse Change-Preview for REST-to-SOAP-to-REST Proxies
Reparsing a WSDL for a REST-to-SOAP-to-REST proxy compared the wrong pair of specs and could report every endpoint as changed even when nothing had changed. The preview now compares only the client-facing contract, matching SOAP operations by soapAction instead of a signature that could never match across the transformation. [See.](/en/develop/api-proxy-creation/multiple-schema-port-wsdl) APNZ-5956

- ### Fixed Policy Drag-and-Drop Preview Showing Only Its Top Half
When dragging a request/response/error policy button to reorder it in the API Proxy Flow screen, the drag preview box was visually cut off. The drag directive is now attached to the outer wrapper element so the full policy box renders under the cursor while dragging. [See.](/en/develop/policies/message-flow-and-policy-management) APNZ-6056

- ### Fixed REST-to-SOAP-to-REST OpenAPI Spec Not Matching Test Console Output
The OpenAPI spec generated for REST-to-SOAP-to-REST proxies produced request/response schemas and examples derived from the raw WSDL components, which did not match what the Test Console produced from the actual transformed sample message. The spec generator now builds the schema directly from the transformed sample message, keeping the published spec, Postman collection, and Test Console output consistent. APNZ-5664

## Version 2026.04.0

:::note

This is a major release that includes the migration of all platform components to Java 25, comprehensive security hardening, and numerous new features.

:::

<span style={{ display: 'inline-block', backgroundColor: '#fecaca', color: '#991b1b', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>REMOVED FEATURES</span>

- **Automatic deletion of audit logs and ACL audit logs via Purge Jobs has been removed.** These logs are critical for compliance and should not be automatically deleted. Manual cleanup scripts are available in the database growth management documentation.
- **The option to disable database writing for audit and login logs in General Settings has been removed.** These logs are now always persisted to the database.

- ### Java 25 Migration
All components of the Apinizer platform (API Manager, Gateway Worker, Cache Server, Integration, API Portal) have been upgraded to Java 25. Virtual thread support, modern JDK features, and performance improvements have been delivered. APNZ-5897

:::warning

**Script Policy — javax → jakarta Auto-Migration:** Starting with this version, Jakarta EE namespaces used in Groovy script policies (e.g. `javax.servlet.*`, `javax.persistence.*`, `javax.xml.bind.*`, `javax.mail.*`, etc.) are automatically converted to their `jakarta.*` equivalents before compilation. Existing scripts will be automatically migrated during the upgrade to this version. **JDK standard library packages** (`javax.crypto.*`, `javax.net.ssl.*`, `javax.xml.parsers.*`, `javax.xml.transform.*`, `javax.script.*`) are **not affected** by this conversion and will continue to work as-is.

:::

## Version 2026.01.01

**Release Date: January 22, 2026**

Kubernetes environment types will be automatically assigned in this release: environments managed through Apinizer will be set as "Managed", others as "Remote".

<span style={{ display: 'inline-block', backgroundColor: '#86efac', color: '#056d4f', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>FEATURED NEW FEATURES</span>

- ### API Manager Application Authorization System 
The authorization system in the Apinizer API Manager application has been renewed. Authorization now works based on the **asset category + action** model. Components with similar functions have been grouped under asset categories (API Management, Connections, Identity and Access Control, etc.) and specific actions (View, Manage, Deploy/Undeploy, Execute, Export/Import) have been defined for each category. This provides a more flexible, detailed, and secure authorization management. [See.](/en/admin/user-access-management/authorization-matrix) 

- ### Cache Server Definition in Namespace Independent from Gateway
The requirement for worker pods that serve as the gateway function in the Apinizer application infrastructure and Hazelcast pods used for cache operations to be in the same namespace has been removed. Worker pods and cache pods can now be managed in separate namespaces. This provides a more flexible infrastructure configuration. [See.](/en/admin/server-management/distributed-cache)

- ### Multi-API Portal Support
Support for managing multiple independent API Portals from a single Apinizer installation has been added. This enables offering customized portal experiences for different business units or customer groups. [See.](/en/api-portal/overview)

- ### Advanced Routing Features
API Gateway routing capabilities have been enriched with **Sticky Session (Session Affinity)**, **Exponential Retry Delay**, and **Active Health Check** mechanisms. This makes traffic management more flexible, resilient, and intelligent. [See.](/en/develop/routing/http-routing)

- ### APIOps Management API Improvements
In line with the vision that all administrative functions of Apinizer can be performed through APIOps Management APIs; full API support has been added for Credential Secrets management, Keys/Keystores operations, and JWK configurations. Additionally, the ability to create and update JWK from certificate, public key, private key, and keystore has been added. [See.](/api-reference/credentials/secrets)

<span style={{ display: 'inline-block', backgroundColor: '#86efac', color: '#056d4f', padding: '0.25rem 0.75rem', borderRadius: '9999px', fontSize: '0.875rem', fontWeight: '500' }}>NEW FEATURES</span>

- ### Upstream Sticky Session (Session Affinity)
Sticky Session support has been added to ensure that clients are always routed to the same backend address. Session state management required for backend structures can now work efficiently through the Gateway with Cookie-based, IP Hash, and Hybrid methods. [See.](/en/develop/routing/sticky-session)

- ### Upstream Active Health Check Mechanism
Active Health Check mechanism has been added that periodically checks the health status of backend services. Backend addresses that are not working correctly are automatically removed from traffic and reintroduced when they recover. [See.](/en/develop/routing/retry-and-failover#active-health-check)

- ### Upstream Exponential Retry Delay
"Exponential Backoff" feature has been added that places increasing delays between retry operations performed in error situations. This provides the necessary time for backend services to recover under heavy load. [See.](/en/develop/routing/retry-and-failover#retry-mechanism)

- ### Environment Diagnostic Screen
Diagnostic screen has been added to monitor the operational status, resource consumption, and health metrics of API Gateway environments. [See.](/en/admin/server-management/environment-diagnostics) APNZ-5665, APNZ-5649
