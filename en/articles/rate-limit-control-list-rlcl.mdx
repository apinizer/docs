In the dynamic world of API development and management, controlling the flow of incoming requests is essential to maintain system stability, ensure fair usage, and prevent abuse.

## Our Inspiration: Access Control Lists (ACL)

The "Access Control List" (ACL) concept, used in security for many years, determines who can access certain resources. It is list-based, target-oriented, and contains clear rules. At Apinizer we thought: "Why not use this logic for rate limiting?"

And **Rate Limit Control List (RLCL)** was born.

## So What Exactly Is RLCL?

![What is RLCL](https://miro.medium.com/v2/resize:fit:700/1*49OVVufJZOyaKiBu5YK7iw.png)

As the name suggests: a system that lets you define rate limit rules in a list-based, targeted way. You no longer have to apply a single limit to every IP, every API key, every user. You can define custom limits such as:

- 5 requests per second for IP `192.168.1.25`,
- 500 requests per minute for API keys starting with `client-abc-*`,
- 10,000 requests per day for specific user IDs

You can also use operator-based flexible target audience conditions (such as equals, contains, starts with, ends with, is in the list) or conditional definitions (e.g. different limits during business hours vs. outside). So the era of "one rule for everyone" is over!

## How Does RLCL Work in Apinizer?

![RLCL workflow](https://miro.medium.com/v2/resize:fit:700/1*dS_UY45iOvurNVqbtjDrhw.png)

It's not complicated; the steps are clear:

1. **A request arrives.**
2. **The identity of the request is determined**: depending on the selected identity source, either the value of a variable (IP, header, user ID, etc.) or the user resolved from authentication is used.
3. **The system checks whether this identity belongs to the target audience** (is it in the identity list, or does it match one of the target audience conditions?).
4. **If it is in the target audience**, the rate limit rule defined specifically for it is applied.
5. **If it is not in the target audience**, the out-of-target action you defined is applied: the request is either blocked, or the general quota reserved for out-of-target requests is applied.
6. **If the limit is exceeded**, the request is rejected (usually with a 429 Too Many Requests error).
7. **Optionally**, rate limit status can be added to response headers (`X-RateLimit-*`).

## Apinizer RLCL Flexibility

Apinizer's Rate Limit Control List solution stands out for its flexibility:

## Time Window Options

RLCL supports both **Fixed Window (FIXED)** and **Sliding Window (SLIDING)** approaches:

- **Fixed Window**: Counters are reset at fixed time intervals (e.g. every hour)
- **Sliding Window**: For any moment, requests in the last X minutes/hours/days are evaluated

This lets you offer solutions suited to different business scenarios.

## Dynamic Identity Definition

Apinizer RLCL offers highly flexible options for determining request identity:

- IP address
- API key
- User ID
- Request headers
- Cookie values
- URL parameters
- Request body fields

You can even use combinations of these! For example, you can define special rules for "requests with a certain user ID coming from a certain IP block."

## Flexible Target Audience Conditions

RLCL lets you define operator-based flexible conditions to determine the target audience. Each condition consists of an operator and a value; the conditions are evaluated with OR logic. For example:

- All IP addresses **starting with** `192.168.`
- API keys that **contain** a certain value
- User IDs that **equal** a certain value
- Identities that **are in** a list of values

This way you can reduce management overhead by defining matching rules instead of individual values.

## Why RLCL? What's the Difference?

**Granular control:** You get clear answers to who can access how much.

**Targeted limiting:** A single bad actor cannot lock down your entire API.

**Flexibility:** Partner, customer, service—whoever it is, you can define different SLAs.

**Transparency:** You can include remaining limit information in responses.

**Manageability:** Definitions are simple and readable.

**Performance:** It runs fast with cache support and doesn't slow down your system.

## Real-World Examples

- **Free vs. Premium Users:** In your freemium service you can apply "100 requests per day" for free users and "50 requests per second" for premium users. Your API stays both accessible and fair.

- **Geography-Based Protection:** You can apply different limits to requests from certain countries or IP blocks. This is especially useful for suspicious traffic sources.

- **DDoS Prevention:** You can quickly detect IPs sending sudden high traffic and apply special restrictions. Thus you isolate the problem without affecting your other users.

- **Partner Management:** You can define specific API usage limits for each business partner. According to everyone's needs—some can make 1000 requests per minute while others make 10 per hour.
