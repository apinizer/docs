## AI Gateway Concept

AI Gateway is the AI layer added to [API Gateway](/en/concepts/core-components/api-gateway) — the same high-performance Data Plane component that already processes your API traffic, not a separate product. It exposes a single, OpenAI-compatible entry point in front of multiple LLM providers, so client applications integrate once and Apinizer handles the rest.

<CardGroup cols={2}>
  <Card title="Single Entry Point" icon="network-wired">

  </Card>

## AI Gateway and API Gateway Relationship

An AI proxy runs through the same kind of request pipeline as any other [API Proxy](/en/concepts/core-concepts/what-is-api-proxy) — security and traffic-management policies apply exactly as they already do — with AI-specific stages (guardrails, RAG, quota, routing to an LLM provider) added to that pipeline rather than replacing it.

## AI Gateway and API Manager Relationship

Configuration and deployment of AI proxies work the same way as any other proxy type:

```
API Manager (Control Plane)
   │
   │ 1. AI Proxy Configuration
   │ 2. Deployment
   │
   ▼
AI Gateway (Data Plane)
   │
   │ 3. Request Processing (guardrails, RAG, quota, routing)
   │ 4. LLM Provider Call
   │ 5. Sending Metrics
   │
   ▼
LLM Provider / Client
```

## Shared Platform Services

AI Gateway does not duplicate platform infrastructure — it reuses what already exists:

<CardGroup cols={2}>
  <Card title="Credentials" icon="key">
    Provider API keys are stored and encrypted the same way as any other Apinizer credential.
  </Card>
  <Card title="Distributed Cache" icon="database">
    Token quotas, throttling, and the semantic cache share the same [Cache Component](/en/concepts/core-components/cache-component) as the rest of the platform.
  </Card>
  <Card title="Analytics & Observability" icon="eye">
    Usage, cost, and guardrail reports run on the same reporting engine used for API traffic.
  </Card>
</CardGroup>
