API Proxy configuration allows you to manage the basic settings and general information of your API Proxy. The Overview tab contains general information about the API Proxy, Client Route configuration, and API Proxy Key management.

## Overview Tab

The **Overview** tab on the API Proxy detail page contains basic configuration information for the API Proxy. This tab consists of the following sections:

## General Information

### Name

The name of the API Proxy. This field is required and must be unique within the project.

**Features:**
* Must be unique within the project
* Spaces and special characters can be used but not recommended
* Should choose a meaningful and descriptive name

### Description

An optional text field describing the purpose and usage scenario of the API Proxy.

**Usage:**
* Describes what the API Proxy is used for
* Used as a reference in management and maintenance processes
* Can be displayed in API documentation

## Client Route

Client Route is the routing information that clients will use to access the API Proxy. At least one option (Path, Method, or Header-based) must be selected for Client Route configuration.

### Relative Paths

Relative URL paths that will be routed to the API Proxy. Multiple paths can be defined.

**Features:**
* Each path must start with "/"
* Must be unique within the application
* Cannot be the beginning of another path

**Example:**
```
/petstoreProxy
/api/v1/users
/products
```

### Methods

HTTP methods allowed for this API Proxy.

**Supported Methods:**
* **GET**: Data reading operations
* **POST**: Creating new records
* **PUT**: Record update (full update)
* **DELETE**: Record deletion
* **PATCH**: Partial update
* **OPTIONS**: CORS preflight requests
* **HEAD**: Getting header information

By default, all methods can be selected.

### Hosts

Host addresses of backend services. Multiple hosts can be added.

**Usage Scenarios:**
* **Load Balancing**: Load distribution among multiple backends
* **Failover**: Switching to another backend if one fails
* **Environment Management**: Different hosts for different environments

**Example:**
```
https://api.example.com
https://api-backup.example.com
```

### Headers

HTTP headers to be added or modified at the API Proxy level. Defined as key-value pairs.

**Usage Scenarios:**
* **Custom Headers**: Adding custom headers
* **Header Transformation**: Transforming header values
* **Security Headers**: Adding security headers

**Example:**
```
X-API-Version: v1
X-Request-ID: ${request.id}
Authorization: Bearer ${token}
```

:::info More Information
For detailed information about Client Route configuration, see the [HTTP Routing](/en/developer/http-routing) page.
:::

## API Proxy Key

The API Proxy Key section is where Public and Secret Keys required for JWT or OAuth2 authorization are managed.

### Key Management

API Proxy Keys are used for:

* **JWT Token Verification**: Verifying JWT tokens
* **OAuth2 Authorization**: OAuth2 authorization operations
* **API Authentication**: API authentication

### Public Key

Public Key is the public key used in token verification operations.

**Features:**
* Can be shared securely
* Used for token verification
* Stored in PEM format

### Secret Key

Secret Key is the secret key used in token signing and encryption operations.

**Features:**
* Must be stored securely
* Used for token signing
* Should never be shared

### Key Generation

To create a new key pair:

1. Click the **Generate Key Pair** button
2. Select the key format (RSA, ECDSA, etc.)
3. Specify the key size
4. Click the **Generate** button

### Key Import

To import an existing key pair:

1. Click the **Import Key** button
2. Paste the Public Key or upload from file
3. Paste the Secret Key or upload from file
4. Click the **Import** button

## Version Information

API Proxy version information is displayed in this section.

### Current Version

The currently active version of the API Proxy.

### Version History

Version history and change records of the API Proxy.

**Information:**
* Version number
* Change date
* User who made the change
* Change description

:::info More Information
For detailed information about version management, see the [Deployment Versioning](/en/developer/deployment-versioning) page.

## Usage Scenarios

### Scenario 1: Simple REST API Proxy

Minimal configuration for a simple REST API Proxy:

* **Name**: `UserServiceAPI`
* **Relative Path**: `/users`
* **Methods**: GET, POST, PUT, DELETE
* **Host**: `https://api.example.com`

### Scenario 2: Load Balanced API Proxy

Multiple hosts for load balancing:

* **Name**: `ProductServiceAPI`
* **Relative Path**: `/products`
* **Hosts**: 
  * `https://api1.example.com`
  * `https://api2.example.com`
  * `https://api3.example.com`

### Scenario 3: JWT Protected API Proxy

API Proxy protected with JWT token verification:

* **Name**: `SecureAPI`
* **Relative Path**: `/secure`
* **API Proxy Key**: JWT Public Key defined
* **Methods**: GET, POST

## Best Practices

### Naming

* **Meaningful Names**: Keep API Proxy names descriptive
* **Consistency**: Maintain consistent naming conventions
* **Short and Clear**: Keep names short and clear

### Client Route

* **Unique Paths**: Ensure paths are unique within the application
* **RESTful Convention**: Follow RESTful conventions
* **Versioning**: Use paths for versioning (e.g., `/api/v1/users`)

### Key Management

* **Security**: Store Secret Keys securely
* **Rotation**: Rotate keys regularly
* **Backup**: Backup keys
