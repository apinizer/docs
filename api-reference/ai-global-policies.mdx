## Endpoints

- [List AI Global Policies](/api-reference/ai-global-policies/list-ai-global-policies) - List all global AI policies in a project (optionally filtered by policy type)
- [Get AI Global Policy](/api-reference/ai-global-policies/get-ai-global-policy) - Get a single global AI policy by name
- [Delete AI Global Policy](/api-reference/ai-global-policies/delete-ai-global-policy) - Delete a global AI policy and strip its embedded copy from every referencing proxy/group
- [Get AI Global Policy Usage](/api-reference/ai-global-policies/get-ai-global-policy-usage) - List the proxies and proxy-groups that reference a global AI policy

## Authentication

All endpoints require authentication using a Personal API Access Token
(`Authorization: Bearer {token}`). JHipster JWTs do not work on APIops.

## Permissions

- Reads (`GET` list / get / usage) require the **`AI_DEVELOPMENT`** asset category with **VIEW**.
- Delete (`DELETE`) requires the **`AI_DEVELOPMENT`** asset category with **MANAGE**.

## Notes

- **No create / update by design** - A global AI policy is not a standalone entity: it is a plain
  `Policy` document with `global=true` living in one of the 11 `PolicyAi*` collections. Creating or
  updating one is done through the generic **export/import pipeline** (`ExportImportResource` /
  `PolicyImportService`), which already covers all 11 AI policy types - not through a bespoke
  create/update endpoint on this resource. This resource exposes only the read / inventory / teardown
  surface (list / get / delete / usage).
- **Name-based** - Policies are resolved by name within the project's `global=true` set across all 11
  collections. Runtime IDs are never required in the path.
- **Polymorphic read** - Reads are serialized through the typed `PolicyDTO` (the same read-path the
  proxy policy endpoints use). A `type` discriminator identifies each policy type, and AI connection
  references (LLM / VectorDB / MCP) are resolved from raw IDs to **names** so the response honours the
  APIops name-based convention.
- **Secrets are never disclosed** - Policy documents do not carry provider secrets; they reference AI
  connection configs by name. The referenced LLM/VectorDB/MCP connection configs hold the
  `@SecretData` fields (`apiKey`, `apiSecret`, `bearerToken`, `clientSecret`), which are never included
  in a policy read (INV-06).
- **Delete safety** - `DELETE` re-verifies project ownership (IDOR guard) and the `global=true`
  invariant before removing the policy; a local (`global=false`) policy is never removable through this
  hub. The removal then fans out via `GlobalPolicyUpdateService` to strip the embedded copy from every
  referencing proxy/group.

## AI global policy types

The optional `type` filter and the `type` discriminator in read responses use these values (either
the string value or the enum name is accepted by the filter):

| Type string | Enum name | Policy |
|-------------|-----------|--------|
| `ai-pii-mask` | `AI_PII_MASK` | PII masking |
| `ai-prompt-guard` | `AI_PROMPT_GUARD` | Prompt-injection guard |
| `ai-token-rate-limit` | `AI_TOKEN_RATE_LIMIT` | Token-based rate limit |
| `ai-semantic-cache` | `AI_SEMANTIC_CACHE` | Semantic response cache |
| `ai-rag-injection` | `AI_RAG_INJECTION` | RAG context injection |
| `ai-mcp-tool-call` | `AI_MCP_TOOL_CALL` | MCP tool call |
| `a2a-agent-call` | `AI_A2A_AGENT_CALL` | A2A agent call |
| `ai-dlp-guard` | `AI_DLP_GUARD` | DLP guard |
| `ai-loop-guard` | `AI_LOOP_GUARD` | Loop guard |
| `ai-topic-guard` | `AI_TOPIC_GUARD` | Topic guard |
| `ai-context-integrity` | `AI_CONTEXT_INTEGRITY` | Context integrity |

An unknown or blank `type` is silently ignored - the query then falls back to scanning all 11
collections.
