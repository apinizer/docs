---
title: "mTLS Authentication Policy"
description: "mTLS (Mutual TLS) Authentication policy authenticates clients using the X.509 certificate presented during the TLS handshake. It resolves the client identity from the certificate's subject or issuer name, optionally validates the certificate chain against the matched Credential's key store, and optionally enforces the Credential's access control list. This policy provides strong authentication based on PKI (Public Key Infrastructure)."
---

#### Request Body Fields

##### operationMetadata

| Field                           | Type    | Required | Default | Description                                    |
|---------------------------------|---------|----------|---------|------------------------------------------------|
| targetScope                     | string  | Yes      | -       | Policy scope: `ALL` or `ENDPOINT`               |
| targetEndpoint                  | string  | No*      | -       | Endpoint path (required if targetScope=ENDPOINT) |
| targetEndpointHTTPMethod        | string  | No*      | -       | HTTP method (required if targetScope=ENDPOINT)  |
| targetPipeline                  | string  | Yes      | -       | Pipeline: `REQUEST`, `RESPONSE`, or `ERROR`     |
| deploy                          | boolean | No       | true    | Whether to deploy after adding policy           |
| deployTargetEnvironmentNameList | array   | No       | []      | List of environment names to deploy to         |
| order                           | integer | No       | null    | Policy execution order (starts from 1)         |

**Enum: targetScope**

- `ALL` - Policy applies to all endpoints
- `ENDPOINT` - Policy applies only to specified endpoint

**Enum: targetPipeline**

- `REQUEST` - Executes in request pipeline (authenticates request)
- `RESPONSE` - Executes in response pipeline
- `ERROR` - Executes in error pipeline

**Enum: targetEndpointHTTPMethod**

- `GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `OPTIONS`, `HEAD`

##### policy

| Field                     | Type    | Required | Default             | Description                                                        |
|---------------------------|---------|----------|---------------------|--------------------------------------------------------------------|
| type                      | string  | Yes      | -                   | Policy type: `policy-auth-mtls`                                     |
| description               | string  | No       | -                   | Policy description                                                  |
| active                    | boolean | No       | true                | Whether policy is active                                            |
| identitySource            | string  | No       | `SUBJECT_OR_ISSUER` | Which certificate name the identity is resolved from                |
| validateCertificate       | boolean | No       | true                | Validate client certificate validity dates and intended usage       |
| validateCertificatesIssuer| boolean | No       | true                | Validate the presented certificate chain against the Credential's key store |
| validateACLForIssuer      | boolean | No       | true                | Validate the Credential's access permission, validity and address restrictions |

### Note

- `identitySource`: Determines which name on the client certificate the identity is resolved from. Allowed values:
  - `SUBJECT` — resolve from the client certificate's own subject CN.
  - `ISSUER` — resolve from the certificate's issuer CN, so all certificates signed by that authority map to a single Credential.
  - `SUBJECT_OR_ISSUER` (default) — try the subject CN first, fall back to the issuer CN.

  The value must be consistent with the username of the Credential you define, otherwise the request is rejected even when the certificate is valid. The identity is read only from the client's own certificate; names of intermediate or root certificates in the chain are never used as an identity.
- `validateCertificate`: When `true`, validates the certificate's validity dates and verifies that it was issued for client authentication purposes.
- `validateCertificatesIssuer`: When `true`, validates the entire certificate chain presented by the client against the trusted certificates in the key store bound to the matched Credential. If no key store is bound to the Credential, the request is rejected.
- `validateACLForIssuer`: When `true`, validates the matched Credential's access permission for this API, its validity period and any address/location restrictions.
- All validations are enabled by default for maximum security.
- This policy is supported only on HTTP/HTTPS API Proxies. It is not supported on gRPC or WebSocket API Proxies; if defined on either, the request fails with a 500 error instead of running the certificate checks.

## Notes and Warnings

- **Identity Resolution**:
  - `identitySource` selects which certificate name the identity is resolved from: `SUBJECT`, `ISSUER` or `SUBJECT_OR_ISSUER` (default)
  - The resolved name must match the username of an enabled Credential in Apinizer
  - The identity is read only from the client's own certificate; intermediate and root certificate names in the chain are never used

- **Certificate Validation**:
  - When `validateCertificate: true`, validates the certificate's validity dates and verifies it was issued for client authentication purposes
  - Certificate must be valid and not expired

- **Trust Validation**:
  - When `validateCertificatesIssuer: true`, validates the entire chain presented by the client against the key store bound to the matched Credential
  - A key store must be bound to the Credential; otherwise the request is rejected
  - Runs before access control, so trust is proven before the client is considered authenticated

- **Access Control**:
  - When `validateACLForIssuer: true`, validates the matched Credential's access permission for this API, its validity period and address/location restrictions
  - The API must be present in the Credential's access list

- **TLS Configuration**:
  - HTTPS and mTLS must be enabled in the environment settings where the policy is applied
  - TLS must terminate on the gateway; if a load balancer or ingress terminates TLS, the client certificate does not reach the gateway
  - Client certificates must be presented during TLS handshake

- **Protocol Support**:
  - Supported only on HTTP/HTTPS API Proxies
  - Not supported on gRPC or WebSocket API Proxies; if defined on either, requests fail with a 500 error rather than the normal 401 authentication failure

- **Certificate Format**:
  - Supports X.509 certificates (PEM, DER formats)
  - Certificate must include public key and signature
  - Certificate chain validation requires intermediate and root CA certificates

- **Security**:
  - mTLS provides strong authentication based on PKI
  - More secure than password-based authentication
  - Requires proper certificate management

- **Performance**: Certificate validation adds minimal overhead to TLS handshake

- **Pipeline**:
  - `REQUEST` pipeline validates client certificate before processing request
  - Authentication failure results in 401 Unauthorized response

- **Error Handling**: Invalid or missing certificates cause authentication to fail

- **Deployment**: Policy changes require deployment to take effect. Set `deploy: true` or deploy manually.
