---
title: "Virtual Keys API"
description: "The Virtual Keys API provides name-based, token-authenticated endpoints for managing AI Gateway virtual keys (provider credential mappings). A virtual key maps an API consumer scope to the vault credential that holds the real LLM provider key, so the consumer never sees the provider secret."
---

## Endpoints

- [List Virtual Keys](/api-reference/virtual-keys/list-virtual-keys) - List all virtual keys in a project
- [Get Virtual Key](/api-reference/virtual-keys/get-virtual-key) - Get a single virtual key by name
- [Create Virtual Key](/api-reference/virtual-keys/create-virtual-key) - Create a virtual key (upsert)
- [Update Virtual Key](/api-reference/virtual-keys/update-virtual-key) - Update an existing virtual key
- [Delete Virtual Key](/api-reference/virtual-keys/delete-virtual-key) - Delete a virtual key
- [Rotate Virtual Key](/api-reference/virtual-keys/rotate-virtual-key) - Rotate the backing credential with a grace window

## Authentication

All endpoints require authentication using a Personal API Access Token
(`Authorization: Bearer {token}`).

## Permissions

- User must have **`AI_DEVELOPMENT`** permission in the project:
  - **VIEW** — for read operations (`GET`)
  - **MANAGE** — for write operations (`POST`, `PUT`, `DELETE`, rotate)

## Notes and Warnings

- **Name-based**: All references are by name — the project and the virtual key are addressed by name, never by runtime id.
- **Upsert**: `POST /{virtualKeyName}/` creates the virtual key, or updates it in place if the name already exists.
- **Credential reference is masked on read**: A virtual key stores no plaintext secret, but `credentialRef` points to the vault `Credential` that holds the real provider key. Read responses return only a boolean presence flag `credentialRefSet` (INV-06) — the backing credential id is never disclosed.
