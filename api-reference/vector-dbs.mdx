## Endpoints

- [List VectorDB Connections](/api-reference/vector-dbs/list-vector-dbs) - List all Vector DB connections in a project
- [Get VectorDB Connection](/api-reference/vector-dbs/get-vector-db) - Get a single Vector DB connection by name
- [Create VectorDB Connection](/api-reference/vector-dbs/create-vector-db) - Create (or upsert) a Vector DB connection
- [Update VectorDB Connection](/api-reference/vector-dbs/update-vector-db) - Update an existing Vector DB connection
- [Delete VectorDB Connection](/api-reference/vector-dbs/delete-vector-db) - Delete a Vector DB connection and undeploy it
- [Test VectorDB Connection](/api-reference/vector-dbs/test-vector-db-connection) - Test connectivity without persisting

## Authentication

All endpoints require authentication using a Personal API Access Token
(`Authorization: Bearer {token}`). JHipster JWTs do not work on APIops.

## Permissions

- **`AI_DEVELOPMENT`** asset category ‚Äî `VIEW` for reads (`GET`), `MANAGE` for writes (`POST` / `PUT` / `DELETE` / `test-connection`)

## Key Concepts

- **Name-based** ‚Äî connections are referenced by `name`, never by runtime id. `POST /{vectorDbName}/` is an **upsert**: it creates the connection, or updates it in place if the name already exists in the project.
- **Polymorphic transport** ‚Äî the connection type discriminator is `_class: "vector-db"` at the top level; the connection details live in a nested `transport` sub-document whose own `family` discriminator (`jdbc` / `http` / `redis`) is chosen by `dbType`.
- **Secret masking on read** ‚Äî the JDBC `password`, HTTP `apiKey`/`apiSecret` and Redis `password` are never returned. Read responses expose only boolean presence flags (`jdbcPasswordSet`, `apiKeySet`, `apiSecretSet`, `redisPasswordSet`).
- **Blank-secret preserve on update** ‚Äî sending a blank secret on `PUT` keeps the stored value (INV-06).
