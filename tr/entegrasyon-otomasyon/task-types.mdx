API Integrator, farklı ihtiyaçlara göre çeşitli task türleri sunar. Bu task türlerini doğru şekilde kullanarak verimli ve güvenilir Task Flow'lar oluşturabilirsiniz.

## Genel Bakış

API Integrator'da şu ana task türleri mevcuttur:

* **Sequential Tasks**: Sıralı çalıştırma
* **Parallel Tasks**: Paralel çalıştırma
* **Decision Tasks**: Koşullu dallanma
* **Action Tasks**: Belirli aksiyonlar (API Call, Database, Email, vb.)

## Sequential Tasks

Sequential task'lar, birbirini takip eden task'ların sırayla çalıştırıldığı task türüdür. Bir task tamamlandıktan sonra bir sonraki task başlar.

### Kullanım Senaryoları

Sequential task'lar şu durumlarda kullanılır:

* Bir task'ın çıktısı diğer task'ın girdisi olduğunda
* Task'ların belirli bir sırada çalışması gerektiğinde
* Bağımlılık zinciri olduğunda

### Örnek

```
Start → Task 1 → Task 2 → Task 3 → End
```

**Akış:**
1. Task 1 çalışır ve tamamlanır
2. Task 2 çalışır ve tamamlanır
3. Task 3 çalışır ve tamamlanır
4. End

### Yapılandırma

Sequential task'lar için özel bir yapılandırma gerekmez. Task'ları sırayla bağlamanız yeterlidir.

**Özellikler:**
* Her task bir önceki task'ın çıktısını kullanabilir
* Bir task başarısız olursa akış durur (error handling yapılandırılmadıysa)
* Task'lar sırayla çalıştığı için toplam süre tüm task'ların sürelerinin toplamıdır

## Parallel Tasks

Parallel task'lar, birden fazla task'ın aynı anda çalıştırıldığı task türüdür. Bu sayede performans artırılır ve toplam çalıştırma süresi azalır.

### Kullanım Senaryoları

Parallel task'lar şu durumlarda kullanılır:

* Task'lar birbirinden bağımsız olduğunda
* Performans optimizasyonu gerektiğinde
* Farklı kaynaklardan veri çekildiğinde

### Örnek

```
Start → Parallel → Task A ──┐
              → Task B ──┼→ Merge → End
              → Task C ──┘
```

**Akış:**
1. Task A, Task B ve Task C aynı anda başlar
2. Tüm task'lar tamamlanır
3. Merge noktasında birleşir
4. End

### Yapılandırma

Parallel task'ları yapılandırmak için:

1. **Parallel Task** ekleyin
2. Paralel çalıştırılacak task'ları bağlayın
3. **Merge Task** ekleyin (tüm task'lar tamamlandıktan sonra birleşme noktası)

**Özellikler:**
* Tüm task'lar aynı anda başlar
* En yavaş task tamamlanana kadar beklenir
* Toplam süre en yavaş task'ın süresidir
* Task'lardan biri başarısız olursa akış durur (error handling yapılandırılmadıysa)

### Merge Stratejileri

Merge noktasında farklı stratejiler kullanılabilir:

* **Wait for All**: Tüm task'lar tamamlanana kadar bekle
* **Wait for Any**: Herhangi bir task tamamlanınca devam et
* **Wait for N**: N task tamamlanınca devam et

## Decision Tasks

Decision task'lar, koşullu dallanma sağlayan task türüdür. Belirli bir koşula göre farklı task'lara yönlendirme yapar.

### Kullanım Senaryoları

Decision task'lar şu durumlarda kullanılır:

* Koşullu iş mantığı gerektiğinde
* Farklı senaryolara göre farklı işlemler yapıldığında
* Veri doğrulama ve yönlendirme gerektiğinde

### Örnek

```
Start → Decision: Check Status
     → [status == "success"] → Success Task → End
     → [status == "error"] → Error Task → End
     → [default] → Default Task → End
```

**Akış:**
1. Decision task çalışır ve koşulu değerlendirir
2. Koşula göre ilgili task'a yönlendirir
3. Seçilen task çalışır
4. End

### Yapılandırma

Decision task'ı yapılandırmak için:

1. **Decision Task** ekleyin
2. Koşulu tanımlayın
3. Her koşul için çıkış bağlantısı oluşturun

**Koşul Örnekleri:**
```
${variable} == "value"
${number} > 100
${status} != "error"
${array.length} > 0
${object.property} == true
```

**Operatörler:**
* `==`: Eşit
* `!=`: Eşit değil
* `>`: Büyük
* `<`: Küçük
* `>=`: Büyük eşit
* `<=`: Küçük eşit
* `&&`: Ve
* `||`: Veya
* `!`: Değil

### Multiple Conditions

Birden fazla koşul kullanabilirsiniz:

```
${status} == "success" && ${count} > 0
${type} == "user" || ${type} == "admin"
!${isDeleted}
```

## Action Tasks

Action task'lar, belirli bir işlemi gerçekleştiren task türleridir.

### API Call Task

HTTP/HTTPS API çağrıları yapar.

**Yapılandırma:**
* **URL**: API endpoint URL'i
* **Method**: HTTP metodu (GET, POST, PUT, DELETE, PATCH)
* **Headers**: Request header'ları
* **Body**: Request body
* **Authentication**: Kimlik doğrulama
* **Timeout**: Timeout süresi

**Örnek:**
```
URL: https://api.example.com/users
Method: POST
Headers: {"Content-Type": "application/json"}
Body: {"name": "${userName}", "email": "${userEmail}"}
```

### Database Task

Veritabanı işlemleri yapar.

**Yapılandırma:**
* **Connection**: Veritabanı bağlantısı
* **Query Type**: Sorgu türü (Select, Insert, Update, Delete, Stored Procedure)
* **Query**: SQL sorgusu
* **Parameters**: Sorgu parametreleri

**Örnek:**
```
Query Type: Select
Query: SELECT * FROM users WHERE id = ?
Parameters: [${userId}]
```

### Email Task

E-posta gönderir.

**Yapılandırma:**
* **To**: Alıcı adresi
* **Subject**: Konu
* **Body**: İçerik
* **Attachments**: Ekler

**Örnek:**
```
To: ${userEmail}
Subject: Hoş geldiniz
Body: Merhaba ${userName}, hoş geldiniz!
```

### Script Task

JavaScript veya Groovy script çalıştırır.

**Yapılandırma:**
* **Language**: Script dili (JavaScript, Groovy)
* **Script**: Script kodu

**Örnek:**
```javascript
var result = {
    fullName: input.firstName + " " + input.lastName,
    timestamp: new Date().toISOString()
};
return result;
```

### Transform Task

Veri dönüşümü yapar.

**Yapılandırma:**
* **Input Format**: Giriş formatı (JSON, XML, CSV)
* **Output Format**: Çıkış formatı (JSON, XML, CSV)
* **Mapping**: Veri eşleme kuralları

**Örnek:**
```
Input: {"fname": "John", "lname": "Doe"}
Mapping: firstName = fname, lastName = lname
Output: {"firstName": "John", "lastName": "Doe"}
```

### Delay Task

Belirli bir süre bekler.

**Yapılandırma:**
* **Duration**: Bekleme süresi (saniye)

**Örnek:**
```
Duration: 5 seconds
```

## Task Kombinasyonları

Farklı task türlerini birleştirerek karmaşık senaryolar oluşturabilirsiniz.

### Örnek Senaryo 1: Sequential + Decision

```
Start → Get User Data → Decision: Is Valid?
     → [Yes] → Update Database → Send Email → End
     → [No] → Log Error → End
```

### Örnek Senaryo 2: Parallel + Merge

```
Start → Parallel → Fetch Data from API 1 ──┐
              → Fetch Data from API 2 ──┼→ Merge Data → Process → End
              → Fetch Data from API 3 ──┘
```

### Örnek Senaryo 3: Nested Decision

```
Start → Decision: User Type?
     → [Admin] → Decision: Has Permission?
              → [Yes] → Execute Admin Task → End
              → [No] → Deny Access → End
     → [User] → Execute User Task → End
```

## Best Practices

### Task Seçimi

* **Sequential**: Bağımlılık varsa kullanın
* **Parallel**: Bağımsız task'lar için kullanın
* **Decision**: Koşullu mantık için kullanın

### Performance

* Mümkün olduğunca parallel task'lar kullanın
* Gereksiz sequential task'lardan kaçının
* Timeout değerlerini uygun şekilde ayarlayın

### Error Handling

* Her task için uygun error handling stratejisi seçin
* Retry mekanizmaları ekleyin
* Fallback task'lar tanımlayın
