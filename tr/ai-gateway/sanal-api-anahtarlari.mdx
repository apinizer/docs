## Genel Bakış

**Sanal API anahtarı**, istemcilerinizin Apinizer AI Gateway'e erişmek için kullandığı anahtardır. İstemci hiçbir zaman gerçek sağlayıcı (OpenAI, Anthropic vb.) API anahtarını görmez — sanal anahtar, arka planda bir veya birden fazla sağlayıcı bağlantısına eşlenir.

```python
from openai import OpenAI

client = OpenAI(
    api_key="sanal-api-anahtarınız",  # gerçek sağlayıcı anahtarı değil
    base_url="https://apinizer-gateway-adresiniz.com/api/ai/v1"
)
```

## Neden Sanal Anahtar?

- **Anahtar sızmaz** — Gerçek sağlayıcı anahtarı yalnızca Apinizer'ın kimlik bilgisi sisteminde şifreli olarak saklanır, istemciye hiç ulaşmaz
- **Merkezi kontrol** — Bir anahtarı iptal etmek veya kapsamını değiştirmek istemci tarafında hiçbir değişiklik gerektirmez
- **Ayrıntılı kapsam** — Aynı sağlayıcı bağlantısı için farklı kullanıcı/ekip/projelere farklı sanal anahtarlar tanımlanabilir

## Kapsam (Scope)

Her sanal anahtar dört kapsamdan birinde tanımlanır:

| Kapsam | Açıklama |
|--------|----------|
| **Kullanıcı** | Tek bir kullanıcıya özel anahtar |
| **Rol** | Belirli bir role sahip tüm kullanıcılar tarafından kullanılabilir |
| **Proje** | Bir proje kapsamındaki tüm kullanıcılar/uygulamalar tarafından kullanılabilir |
| **Takım** | Bir takımın tüm üyeleri tarafından kullanılabilir |

## Anahtar Oluşturma

<Steps>
  <Step title="LLM Sağlayıcıları Sayfasına Gidin">
    **Yönetim** → **API Gateway** → **LLM Sağlayıcıları** → **Sanal API Anahtarları** bölümüne gidin.
  </Step>

  <Step title="Sağlayıcı Bağlantısını Seçin">
    Sanal anahtarın eşleneceği sağlayıcı bağlantısını (veya bağlantılarını) seçin.
  </Step>

  <Step title="Kapsamı Belirleyin">
    Anahtarın kapsamını seçin: Kullanıcı, Rol, Proje veya Takım.
  </Step>

  <Step title="Kaydet">
    **Kaydet**'e tıklayın. Oluşturulan sanal anahtar istemcilere dağıtılmaya hazırdır.
  </Step>
</Steps>

## Kesintisiz Rotasyon

Bir sanal anahtarı döndürürken (örn. periyodik güvenlik politikası gereği) istemci tarafında kesinti yaşanmaz:

1. Yeni anahtar oluşturulur ve öncelikli hale getirilir
2. Eski anahtar, tanımlanan bir **grace period** boyunca geçerliliğini korur
3. İstemciler bu süre içinde yeni anahtara geçiş yapar
4. Grace period sonunda eski anahtar devre dışı kalır

:::tip
Rotasyonu istemci tarafındaki dağıtım takvimine göre planlayın — grace period, eski ve yeni anahtarın bir arada geçerli olduğu güvenli geçiş penceresidir.
:::

## Kimlik Bilgisi Sistemiyle İlişki

Sanal anahtarların eşlendiği gerçek sağlayıcı kimlik bilgileri, Apinizer'ın genel kimlik bilgisi sistemi üzerinden yönetilir. Detaylar için [Kimlik Bilgileri](/tr/admin/identity-management/kimlik-bilgileri) sayfasına bakın.
