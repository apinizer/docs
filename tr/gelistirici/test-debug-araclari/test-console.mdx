Test Konsolu üzerinden Postman ya da SOAP UI gibi başka bir araç kullanmadan, SOAP ve REST API'ler test edilir ve test kayıtları yönetilebilir.

**Test Konsolu içeren görsele aşağıda yer verilmiştir:**

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-1.png" alt="Test Konsolu" width="800" />

Test Konsolu 3 bölümden oluşmaktadır;

1. İstek (Request): Bu bölümde istek hazırlama işlemleri yönetilir.
2. Yanıt (Response): Bu bölümde yanıta ait sonuç işlemleri yönetilir.
3. Test Kayıtları: Bu bölümde test geçmişi ve grup olarak kaydedilmiş testler görülür.

## İstek (Request)

### İstek Adresini Oluşturma

İstek adresi oluşturulmanın 2 yolu bulunur;

* Test etmek istenilen endpoint manuel olarak yazılabilir.
* **Test Helper** arayüzünden Apinizer üzerindeki bir API Proxy test edilebilir ya da API tanım dosyası üzerinden istek adresi oluşturulabilir.

**Test Helper ile İstek Adresini Oluşturma**

Aşağıdaki görselde seçili butona tıklandığında Test Helper dialogu açılır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-2.png" alt="Test Helper butonu" width="600" />

**Test Helper ayarlarını içeren görsele aşağıda yer verilmiştir:**

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-3.png" alt="Test Helper ayarları" width="600" />

Test Helper konfigürasyonu için kullanılan alanlar aşağıdaki tabloda görülmektedir.

| Alan | Açıklama |
|------|----------|
| **İstek Tipi** <em>(Request Type)</em> | Proxy, Proxy Group, Swagger, OpenApi, WADL, WSDL seçeneklerinden biri seçilerek test edilecek API tipi belirlenir. Bu tipler:<br/><br/>**Proxy, Proxy Group**<br/><br/>Bu tipler, Apinizer üzerinde seçili projenizde bulunan API Proxy'ler test edilmek istendiğinde seçilir.<br/><br/>Ayrıca, seçili API Proxy'nin deploy edildiği ortamlardan (environments) test edilir.<br/><br/>**Swagger, OpenApi, WADL, WSDL**<br/><br/>Bu tipler, API tanım dosyası üzerinden test yapılmak istendiğinde seçilir. |
| **Proxy Grup Listesi** <em>(Proxy Group List)</em> | Aynı proje içindeki test edilebilecek API Proxy grupları listelenir. |
| **Proxy Listesi** <em>(Proxy List)</em> | Aynı proje içindeki test edilebilecek API Proxy'ler listelenir.<br/><br/>Proxy'nin test edilebilmesi için deploy edilmiş olması gerekir. |
| **Servis URL** <em>(Service URL)</em> | Test edilecek API'nin servis URL'idir. URL vasıtasıyla parse işlemi yapılır ve API'nin uç noktaları oluşturulur. |
| **Uç Noktalar** <em>(Endpoints)</em> | API'de bulunan uç noktalar arasından test edilecek uç nokta seçilir. |
| **Ortam Listesi** <em>(Environment List)</em> | Proxy'nin test edileceği ortam (environment) seçilir. |

<Info>
Oluşturulan uç noktanın eğer varsa parametre listesi ve örnek istek gövdesi (request body) de Apinizer tarafından oluşturulmaktadır.
</Info>

#### WS Security İmzala

İsteğin gövdesine eklenecek veri raw olarak seçildiğinde WS Security İmzala işlemi yapılabilir.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-9.png" alt="WS Security İmzala" width="600" />

**WS Security İmzala ekranını içeren görsele aşağıda yer verilmiştir:**

WS Security İmzalama tuşuna basıldığında aşağıdaki konfigürasyonla test konsolundaki veri imzalanır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-10.png" alt="WS Security İmzala ayarları" width="600" />

WS Security İmzala konfigürasyonu için kullanılan alanlar aşağıdaki tabloda görülmektedir.

| Alan | Açıklama |
|------|----------|
| **Private Key** <em>(Private Key)</em> | İmzalama için kullanılacak Private Key değerinin PKCS#8 PEM formatındaki metinsel halidir. |
| **Sertifika** <em>(Certificate)</em> | İmzalama için kullanılacak sertifikanın PKCS#8 PEM formatındaki metinsel halidir. |
| **Signature Algorithm** <em>(Signature Algorithm)</em> | İmzalama algoritması bilgisidir. |
| **Signature Canonicalization** <em>(Signature Canonicalization)</em> | İmza kanonikleştirme bilgisidir. |
| **Digest Algorithm** <em>(Digest Algorithm)</em> | İmzada kullanılacak digest algoritmasıdır. |
| **Key Identifier Type** <em>(Key Identifier Type)</em> | İmza anahtarının mesajın neresine nasıl yerleştirileceği bilgisidir. |
| **Use Single Certificate** <em>(Use Single Certificate)</em> | İmzalamanın tek sertifika ile mi yapılacağı bilgisidir. |
| **Must Understand** <em>(Must Understand)</em> | WS-Security başlığında "Must Understand" değerinin ne olması gerektiğini belirtir. |
| **WS Encryption Part** <em>(WS Encryption Part)</em> | Verinin nerelerinin imzalanacağı bilgisidir. Birden fazla değer girilebilir.<br/><br/>İmzalanacak XML elemanı adı, XML elemanının namespace bilgisi ve Element olarak mı İçerik olarak mı imzalanacağı bilgisidir. |

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-12.png" alt="JSON İmzala" width="600" />

**JSON İmzala ayarlarını içeren görsele aşağıda yer verilmiştir:**

JSON imzalama tuşuna basıldığında aşağıdaki konfigürasyonla test konsolundaki veri imzalanır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-11.png" alt="JSON İmzala ayarları" width="600" />

JSON imzala konfigürasyonu için kullanılan alanlar aşağıdaki tabloda görülmektedir.

| Alan | Açıklama |
|------|----------|
| **Algoritma** <em>(Algorithm)</em> | İmzalama yapılacak algoritma bilgisidir |
| **Private Key** <em>(Private Key)</em> | İmzalama için kullanılacak Private Key değerinin PKCS#8 PEM veya JWK formatındaki metinsel halidir. |
| **Issue Time Ekle** <em>(Add Issue Time)</em> | Oluşturulacak imzanın içine oluşma zamanı eklenecekse bu seçenek aktifleştirilmelidir. |
| **Expiration Time Ekle** <em>(Add Expiration Time)</em> | Oluşturulacak imzanın içine tükenme zamanı eklenecekse bu seçenek aktifleştirilmelidir. |
| **Expiration Time Value** <em>(Expiration Time Value)</em> | İmzanın ne kadar süre geçerli olacağı bilgisidir |
| **Expiration Time Unit** <em>(Expiration Time Unit)</em> | İmzanın geçerli olma süresinin zaman birimidir. |
| **JWT Headers** <em>(JWT Headers)</em> | İmzalanacak verinin başlık kısmına ekstra başlık bilgisi eklenmek istenirse bu alana JSON formatında eklenebilir. |
| **Payload** <em>(Payload)</em> | İmzalanacak verinin gövde kısmına ekstra bilgi eklenmek istenirse bu alana JSON formatında eklenebilir. |

### Teyit (Assertions) Oluşturma

Teyit tabı, istek gönderirken dönecek olan yanıtla ilgili zaman aşımı, HTTP durum kodu, gövde, JsonPath/Xpath ile gövdedeki değeri sorgulayarak beklenen değerlerle doğrulanmasına izin verir.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-13.png" alt="Teyit Oluşturma" width="600" />

Teyit konfigürasyonu için kullanılan alanlar aşağıdaki tabloda görülmektedir.

| Alan | Açıklama |
|------|----------|
| **Zaman Aşımı'nı Teyit İçin Kullan** <em>(Use Timeout For Assertion)</em> | Eğer aktifleştirilirse, isteğin zaman aşımı değeri kullanarak doğrulama yapılır. |
| **Sonuç Durum Kodu** <em>(Assert Result Status Code)</em> | Durum kodu bazında doğrulama yapmak için aktifleştirilir. |
| **Beklenen Durum Kodu** <em>(Expected Status Code)</em> | Beklenen durum kodunun değeri yazılır. |
| **Sonuç Gövdesi** <em>(Assert Result Body)</em> | Yanıtın gövdesini doğrulamak için aktifleştirilir. |
| **Beklenen Sonuç Gövdesi** <em>(Expected Result Body)</em> | Beklenen yanıt gövdesinin değeri yazılır. |
| **XPath Sonucu** <em>(Assert Result XPath)</em> | Yanıtın gövdesinin içindeki bir değeri doğrulamak için aktifleştirilir. Yanıtın gövde değerine ait veri tipi XML formatında olmalıdır. |
| **Beklenen XPath** <em>(XPath Expression)</em> | Yanıt gövdesinde doğrulanmak istenen değeri tutan anahtar bilgisi için XPath sorgusu yazılır. |
| **Beklenen XPath Sonucu** <em>(Expected Xml Result)</em> | Yanıt gövdesinde doğrulanmak istenen XPath bilgisininde beklenen sonuç yazılır. |
| **JsonPath Sonucu** <em>(Assert Result JsonPath)</em> | Yanıtın gövdesinin içindeki bir değeri doğrulamak için aktifleştirilir. Yanıtın gövde değerine ait veri tipi JSON formatında olmalıdır. |
| **Beklenen JsonPath** <em>(JsonPath Expression)</em> | Yanıt gövdesinde doğrulanmak istenen değeri tutan anahtar bilgisi için JsonPath sorgusu yazılır. |
| **Beklenen JsonPath Sonucu** <em>(Expected JsonPath Result)</em> | Yanıt gövdesinde doğrulanmak istenen JsonPath bilgisininde beklenen sonuç yazılır. |

## Geçmiş (History)

Test Konsolu'nda test edilen her isteğin (request) geçmişi saklanır. Testin geçmişi, test edildikleri günlere göre kategorilere ayrılır.

Testin loguna ait linke tıklandığında, istek tekrar test edilebilir.

Test geçmişini içeren görsele aşağıda yer verilmiştir:

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-21.png" alt="Test Geçmişi" width="400" />

### Test Geçmişini Silme İşlemleri

* Tüm geçmiş bilgilerini silmek için **Hepsini Sil (Clear All)** linkine tıklanır.
* Gün bazlı geçmişi silmek için ilgili gün bilgisinin detay menüsünden silinir.
* Sadece geçmişi silmek için ilgili geçmiş bilgisinin detay menüsünden silinir.

## Koleksiyon

Koleksiyon, testlerin tekrar kullanılabilmesi için test kayıtlarının bir grup çatısı altında yapılandırılarak oluşturulmasıdır.

Her koleksiyon içinde birden fazla koleksiyon bulunabilir.

### Testi Bir Koleksiyona Kaydetme

Testinizi tamamladıktan sonra kayıt etmek için görseldeki işaretlenen **Kaydet (Save)** butonuna tıklanır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-22.png" alt="Testi Koleksiyona Kaydetme" width="600" />

Sonrasında açılan **Koleksiyon Listesi (Collection List)** adlı dialogda testin atanacağı koleksiyonun linkine tıklanır.

Apinizer başlangıç için 'Default Collection' adlı bir koleksiyonu oluşturmaktadır.

Koleksiyonunu seçtikten sonra teste bir ad vererek kaydedilir. Varsayılan test kaydının ismi, test ettiğiniz endpoint URL bilgisi olarak belirlenmiştir.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-23.png" alt="Koleksiyon Listesi" width="400" />

### Koleksiyon Bazlı İşlemler

Koleksiyon üzerinde yapılabilecek işlemlere, aşağıdaki görseldeki koleksiyonun detay menüsü üzerinden yapılır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-24.png" alt="Koleksiyon Bazlı İşlemler" width="300" />

**Yeniden Adlandır (Rename):** İlgili koleksiyonun adını değiştirmek için kullanılır. Tıklandığında çıkan ekrana yeni ad verilir ve kaydedilir.

**Yeni Koleksiyon Ekle (Add Collection):** İlgili koleksiyonun içine yeni bir koleksiyon eklenmek için kullanılır.

**Koleksiyonu Sil (Delete Collection):** İlgili koleksiyonu silmek için kullanılır. Eğer bir koleksiyonu silinirse, ilgili koleksiyona bağlı bütün kayıtlar silinecektir.

### Kayıt Bazlı İşlemler

Kayıt üzerinde yapılabilecek bir çok işlemlere, aşağıdaki görseldeki detay menüsü üzerinden ulaşılır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-25.png" alt="Kayıt Bazlı İşlemler" width="300" />

**Rename:** İlgili testin adını değiştirmek için kullanılır. Tıklandığında çıkan ekrana yeni ad verilir ve kaydedilir.

**Duplicate:** İlgili testin bir kopyasının oluşturulması için kullanılır.

**Create Monitor:** İlgili testi kullanarak bir Çalışma Süresi Monitörü (Uptime Monitor) oluşturmak için kullanılır. Detaylı bilgi almak için [Uptime Monitor](/tr/gelistirici/monitoring-alerting/uptime-monitor) sayfasına tıklayınız.

**Delete Test:** İlgili testi silmek için kullanılır.

**Farklı Kaydetme (Save As):** Kaydedilmiş bir testi başka bir koleksiyona kaydedebilmek için görselde işaretlenen butona tıklanır.

<img src="/images/gelistirici/test-debug-araclari/test-console/test-console-26.png" alt="Farklı Kaydetme" width="600" />
