## API Proxy Oluşturma Adımları

<Steps>
  <Step title="Menüden API Proxy'ler Sayfasına Gidin">
    Ana menüde **Geliştirme → API Proxy'ler (Development → API Proxies)** ögesi seçilir.
  </Step>
  
  <Step title="Yeni API Proxy Oluşturun">
    Açılan arayüzde sağ üstteki **+Create** butonuna tıklanır.
  </Step>
  
  <Step title="API Proxy Tipini Seçin">
    Yeni oluşturulacak API Proxy için aşağıdaki seçenekler görülür. Uygun tip seçilir.
  </Step>
</Steps>

## API Proxy Tipleri

Oluşturulacak API Proxy için uygun tip seçilir. Mevcut API Proxy tipleri aşağıda görselleştirilmiştir.

<CardGroup cols={2}>
  <Card title="Swagger 2.x" icon="file-code">
    Swagger 2.x tanımından REST API Proxy oluşturmak için kullanılır. Detaylı bilgi için [Swagger 2.0 spesifikasyonu](https://swagger.io/docs/specification/2-0/what-is-swagger/) sayfasına bakabilirsiniz.
  </Card>
  
  <Card title="WSDL (SOAP)" icon="file-code">
    WSDL tanımından SOAP API Proxy oluşturmak için kullanılır. Detaylı bilgi için [WSDL](https://en.wikipedia.org/wiki/Web_Services_Description_Language) sayfasına bakabilirsiniz.
  </Card>
  
  <Card title="OpenAPI/Swagger 3.0.x" icon="file-code">
    OpenAPI/Swagger 3.0.x tanımından REST API Proxy oluşturmak için kullanılır. Detaylı bilgi için [OpenAPI spesifikasyonu](https://swagger.io/docs/specification/about/) sayfasına bakabilirsiniz.
  </Card>
  
  <Card title="Reverse Proxy" icon="arrow-right-arrow-left">
    HTTP Reverse Proxy oluşturmak için kullanılır. Gelen istekleri, ilgili metot/endpoint olup olmadığına bakmaksızın Backend API'ye iletir ve Backend API'den dönen yanıtları istemciye geri döndürür. Tanım dosyasına gerek yoktur ve Backend API'nin metotlarını/endpointlerini tek tek tanımlamaya gerek yoktur. Backend API'nin kök bağlamını (root context) vermek yeterlidir.
  </Card>
  
  <Card title="No-Spec API" icon="code">
    Daha önce code-first yaklaşımıyla geliştirilmiş ve tanım dosyası olmayan bir API için, metotları/endpointleri manuel olarak tanımlama ve API Proxy oluşturma imkanı sağlar. Apinizer, tanımlara göre Swagger veya OpenAPI formatlarında API tanım dosyalarını otomatik olarak oluşturur.
  </Card>
  
  <Card title="gRPC" icon="network-wired">
    Gelen gRPC isteklerini Backend gRPC Sunucusuna ileten ve Backend gRPC Sunucusundan dönen yanıtları istemciye geri döndüren gRPC Proxy oluşturmak için kullanılır. Tanım dosyasına gerek yoktur, ancak proto dosyası kullanılabilir.
  </Card>
  
  <Card title="WebSocket" icon="network-wired">
    WebSocket Proxy oluşturmak için kullanılır. Detaylı bilgi için [WebSocket API Proxy Oluşturma](/tr/gelistirici/api-proxy-olusturma/websocket-api-proxy-olusturma) sayfasına bakabilirsiniz.
  </Card>
  
  <Card title="Connector" icon="plug">
    Konnektörlerden API proxy'si oluşturma yeteneği sağlar. Bu özellik, HTTP'yi farklı protokollere (Kafka, RabbitMQ, ActiveMQ, Mail... vb gibi) dönüştürme imkanı sağlar. Konnektörler hakkında daha fazla bilgi edinmek için [Connector API Proxy Oluşturma](/tr/gelistirici/api-proxy-olusturma/connector-api-proxy-olusturma) sayfasını ziyaret edin.
  </Card>
</CardGroup>

## API Tanım Belgesi Kaynağı

Ayrıca seçeneklerin altlarında bazı ek seçenekler olduğu da görülmektedir. API tanım belgesini nasıl sağlayacağınızı seçebilirsiniz:

<CardGroup cols={3}>
  <Card title="URL Gir" icon="link">
    Kendisi için API Proxy oluşturulacak API'nin API Tanım belgesinin erişilebilir olduğu URL bilinmektedir. Bu URL belirtilerek API Proxy oluşturulabilir.
    
    **Örnekler:**
    - `https://petstore.swagger.io/v2/swagger.json`
    - `http://www.dneonline.com/calculator.asmx?WSDL`
  </Card>
  
  <Card title="Dosya Yükle" icon="upload">
    Kendisi için API Proxy oluşturulacak API'nin API Tanım belgesi bir dosya olarak kayıtlı halde kullanıcıda ise bu dosya yüklenerek API Proxy oluşturulabilir.
  </Card>
  
  <Card title="API Tanım Belgesi Seç" icon="file-lines">
    Kendisi için API Proxy oluşturulacak API için, [API Tanım Belgesi Editörü (Spec Designer)](/tr/gelistirici/api-tasarimi/spec-designer) ile önceden oluşturulmuş olan bir API Tanım Belgesi kullanılarak API Proxy oluşturulabilir.
  </Card>
</CardGroup>

## API Proxy Oluşturma Formu

Oluşturulacak API Proxy için uygun olan tip ve seçenek seçilir. Bu örnek için Swagger 2.x tipinin **URL Gir (Enter URL)** bağlantısına tıklanır.

<AccordionGroup>
  <Accordion title="Spec Authorization Kullan (Use Spec Authorization)">
    Bazı tanım dosyaları, içeriklerinin görüntülenebilmesi için istemcinin yetkilendirilmiş olduğunu kontrol eder. Bunun için HTTP isteği içerisinde kimlik doğrulama verilerinin gönderilmesi gerekir. Eğer erişilecek URL'deki tanım dosyası böyle bir dosya ise, bu kutu işaretlenerek kimlik bilgilerini girilir.

    **Spec Authorization Kullan** seçeneği işaretlenirse yetkilendirme için gerekli olan anahtarlar eklenir.
  </Accordion>
  
  <Accordion title="URL ile API Tanım Belgesi Yükleme">
    **URL** alanına tanım dosyasına erişilecek adres yazılıp sağ taraftaki **Parse** tuşuna tıklanır.

    URL alanında verilen adresteki Swagger dosyası ayrıştırılarak API'nin bilgileri görüntülenir.
  </Accordion>
</AccordionGroup>

### API Proxy Konfigürasyon Alanları

Gelen arayüzdeki alanlar aşağıda kategorilere ayrılarak açıklanmıştır.

<AccordionGroup>
  <Accordion title="Temel Bilgiler">
    <CardGroup cols={2}>
      <Card title="Adı (Name)" icon="tag">
        API Proxy'ye kullanıcı tarafından verilen addır. Arayüzlerde bu ad görünür. Proje bazında benzersiz olmalıdır.
      </Card>
      
      <Card title="Açıklama (Description)" icon="file-lines">
        API Proxy'nin tanım bilgisidir. Girilmesi isteğe bağlıdır.
      </Card>
    </CardGroup>
  </Accordion>
  
  <Accordion title="Routing ve Yönlendirme">
    <CardGroup cols={2}>
      <Card title="Routing Addresses" icon="network-wired">
        Backend API'nin erişim adresleri. Status checkbox ile aktif/pasif edilebilir. Birden fazla adres load balancing için eklenebilir.
      </Card>
      
      <Card title="Client Route" icon="route">
        İstemcilerin API Proxy'e erişim için kullanacağı yönlendirme yöntemi (Path, Method veya Header bazlı). Detaylı bilgi için [API Proxy Client Route Kullanım Kılavuzu](/tr/gelistirici/api-proxy-konfigurasyonu/overview) sayfasına bakabilirsiniz.
      </Card>
      
      <Card title="Relative Paths" icon="link">
        API Proxy'nin erişime açılan adresinin bir parçası. Gelen isteklerin uygun API Proxy'e yönlendirilmesi için kullanılır.
        
        **Örnek:** Apinizer adresi: `https://demo.apinizer.com`, Relative Path: `/petstoreProxy`, Ortam adresi: `/apigateway`, Endpoint: `/findByStatus`, Tam erişim adresi: `https://demo.apinizer.com/apigateway/petstoreProxy/findByStatus`
      </Card>
      
      <Card title="Methods" icon="list">
        API Proxy'nin desteklediği HTTP metodları (GET, POST, PUT, DELETE, PATCH, OPTIONS, HEAD).
      </Card>
    </CardGroup>
  </Accordion>
  
  <Accordion title="Backend ve Ağ Ayarları">
    <CardGroup cols={2}>
      <Card title="Hosts" icon="server">
        Backend servislerin host adresleri. "Add a host" ile yeni adres eklenebilir.
      </Card>
      
      <Card title="Headers" icon="heading">
        API Proxy seviyesinde eklenecek HTTP header'ları. "Add a header" ile yeni header eklenebilir.
      </Card>
      
      <Card title="Backend API Versiyon" icon="code-branch">
        API Proxy'e tanımlanan Backend API Versiyonudur.
      </Card>
    </CardGroup>
  </Accordion>
  
  <Accordion title="Organizasyon ve Paylaşım">
    <CardGroup cols={2}>
      <Card title="Kategori Listesi" icon="folder">
        API Proxy'lerin yönetiminde kolaylık sağlamak üzere kategoriler oluşturulabilir. Kategori listesi, API Proxy'nin hangi kategorilere ait olduğunu belirlemek için kullanılır.
      </Card>
      
      <Card title="Kullanım Şekli (Usage)" icon="users">
        Backend API'nin kaynağının kim olduğunu belirtir. Alabileceği değerler:
        - **Üretici (Publisher)**: Backend API, firma/kurumun kendisi tarafından açılmıştır.
        - **Tüketici (Consumer)**: Backend API, başka bir firma/kuruma aittir. O API'ye erişmek için API Proxy oluşturulmaktadır.
        - **Üretici ve Tüketici (Publisher & Consumer)**: Başka bir firma/kuruma ait olan bir Backend API için oluşturulan API Proxy, 3. taraflara da kullandırılmak üzere oluşturulmaktadır.
      </Card>
      
      <Card title="Paylaşım Tipi" icon="share">
        Backend API'nin nasıl paylaşılacağı belirtilir. Alabileceği değerler: **Dış (External)**, **İç (Internal)**, **Dış ve İç (External and Internal)**
      </Card>
    </CardGroup>
  </Accordion>
</AccordionGroup>

### Relative Path Özellikleri

<Warning>
  Relative Path için dikkat edilmesi gerekenler:
  - Uygulama bazında tekil olmalıdır
  - Başka bir path'in başlangıcı olamaz
  - Proje ayarlarında Enable Relative Path aktifse, proje path'i otomatik önüne eklenir
</Warning>

<AccordionGroup>
  <Accordion title="gRPC Relative Path Formatı">
    Eğer API Proxy tipi **gRPC** ise bu değer proto dökümanındaki `/packagename.servicename` şeklinde olmalıdır. Örneğin aşağıdaki gibi bir proto dosyasına sahip ise relative path değeri `/hello.HelloService` şeklinde olmalıdır.

    ```proto
    syntax = "proto3"
    package hello;
    service HelloService {
    }
    ```
  </Accordion>
  
  <Accordion title="WebSocket Relative Path">
    Eğer API Proxy tipi **WebSocket** ise relative path değerinin nasıl girilmesi gerektiği ile ilgili [WebSocket API Proxy Oluşturma](/tr/gelistirici/api-proxy-olusturma/websocket-api-proxy-olusturma) sayfasını ziyaret edebilirsiniz.
  </Accordion>
</AccordionGroup>

## API Proxy Kaydetme

<Steps>
  <Step title="Alanları Kontrol Edin">
    Tüm gerekli alanların doldurulduğundan emin olun.
  </Step>
  
  <Step title="API Proxy'yi Kaydedin">
    Sağ üstteki **Kaydet (Save)** tuşuna tıklayarak API Proxy kaydedilir.
  </Step>
  
  <Step title="Sonraki Adımlar">
    API Proxy kaydedildikten sonra, API Proxy'nin konfigürasyonu, politikaları ve diğer ayarları yapılabilir.
  </Step>
</Steps>

<Check>
  API Proxy başarıyla oluşturuldu! Artık politikalar ekleyebilir, konfigürasyon yapabilir ve deploy edebilirsiniz.
</Check>
